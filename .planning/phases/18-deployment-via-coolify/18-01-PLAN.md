       - All 6 services with `image: ghcr.io/zerouser-cloud/email-platform-<service>:${IMAGE_TAG:-latest}`
       - NO `build:` directives (server pulls pre-built images)
       - NO `env_file:` directives (Coolify injects env vars)
       - NO `depends_on:` for infrastructure (infra is separate Coolify resources)
       - NO `networks:` block (Coolify manages networking, custom networks cause Gateway Timeout)
       - NO `include:` directive (infrastructure is Coolify-managed)
       - Gateway: `ports: ["3000:3000"]` (Traefik routes externally via Coolify domain config)
       - Other services: `expose:` with their HTTP and gRPC ports
       - All services: `restart: unless-stopped`, `init: true`
       - All services: healthcheck using `wget -qO/dev/null http://127.0.0.1:<port>/health/live` with interval 10s, timeout 5s, retries 5, start_period 10s
       - Service ports from catalog: gateway 3000, auth 3001/50051, sender 3002/50052, parser 3003/50053, audience 3004/50054, notifier 3005

    2. Add deploy job to `.github/workflows/docker-build.yml`:
       - New job `deploy` that runs after `build-and-push` completes (`needs: build-and-push`)
       - Uses `curl` to trigger Coolify webhook: `curl --request GET '${{ secrets.COOLIFY_WEBHOOK }}' --header 'Authorization: Bearer ${{ secrets.COOLIFY_TOKEN }}'`
       - Runs on ubuntu-latest
       - This triggers Coolify to pull latest images and redeploy the compose stack

    CRITICAL anti-patterns to avoid in docker-compose.prod.yml:
    - No `networks:` block at all (Coolify pitfall 1)
    - No `env_file:` (Coolify pitfall 6)
    - No hardcoded hostnames for infrastructure (Coolify pitfall 3)
