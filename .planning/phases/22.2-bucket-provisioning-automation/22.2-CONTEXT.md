# Phase 22.2: bucket-provisioning-automation - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

**SCOPE CHANGE from original ROADMAP.md goal.** Оригинальный goal был: _"Every bucket declared by a per-bucket module is automatically checked and created on service boot"_ — т.е. кодовый auto-provisioning с env flag. В ходе discuss-phase этот подход отклонён.

**Новый scope (docs-only):** Документировать полную процедуру создания S3 bucket'ов во всех четырёх окружениях развёртывания как runbook, который любой оператор может выполнить без предварительных знаний. **Нулевые изменения в коде приложения, docker-compose, env-схемах.** Существующий `HeadBucket` health check из Phase 22 остаётся единственным механизмом верификации.

**Deliverable:** один markdown файл `docs/runbooks/bucket-provisioning.md`.

**Out of scope (explicit):**
- Любой кодовый provisioning service
- Любые новые env vars (STORAGE_PROVISION_ENABLED и т.п.)
- docker-compose init containers (например `minio-init`)
- pnpm helper scripts для local setup
- Изменения в `apps/`, `packages/`, `infra/`
- Исправление gateway gRPC health gap (документируется, но не чинится)

**Scope change implications:**
- `ROADMAP.md` phase 22.2 entry должен быть переписан (goal, requirements list, success criteria)
- `REQUIREMENTS.md` секция "S3 Bucket Provisioning" (SPRV-01..05) должна быть переписана под docs-oriented requirements
- `PROJECT.md` Active requirements должен быть проверен на SPRV упоминания
- Phase 22.3 dependency остаётся валидной в новой формулировке: _"перед smoke tests выполни runbook, создай buckets"_
- Обновления meta-docs — **первая задача в plan-phase**, не часть discuss-phase

## Implementation Decisions

### Core scope decision
- **D-01:** Phase 22.2 становится docs-only. Единственный deliverable — markdown runbook. Ноль кода, ноль изменений в инфраструктуре.
- **D-02:** Обоснование выбора (фиксируется здесь и дублируется в самом runbook'е как "Rationale section"):
  - **12-factor:** приложение не управляет infrastructure state. Создание buckets — infrastructure concern, не application concern.
  - **Garage key bindings:** Garage разделяет bucket и access key как отдельные сущности. `CreateBucket` через S3 API создаёт bucket, но **не** создаёт key binding (`garage key allow --read --write <bucket> <key>`). Приложение создавшее bucket через S3 API получит недоступный bucket — own access key не сможет читать/писать. Верификация этого поведения может быть сделана позже в отдельной фазе research, но для целей 22.2 предполагаем несовместимость и не идём по S3 API пути на Garage.
  - **Minimal permissions на prod:** prod credentials сервиса обычно write-only на уровне объектов, без `s3:CreateBucket` или даже `s3:HeadBucket` на уровне сервиса. Любой кодовый provisioning упадёт на AccessDenied, не на NotFound.
  - **Unified approach во всех окружениях:** "код который работает только локально (flag=true) и дремлет на prod (flag=false)" — code smell. Лучше нулевая автоматизация везде, чем разное поведение по окружениям.
  - **Safety против silent misconfig:** при неправильном `STORAGE_BUCKET` name кодовый provisioning тихо создаёт wrong bucket в prod и сервис работает с неправильным хранилищем. Manual approach + HeadBucket в readiness — опечатка моментально ловится readiness DOWN.
- **D-03:** Приложение остаётся полностью agnostic к bucket lifecycle. Использует уже созданные buckets через готовый `S3HealthIndicator` (Phase 22). `HeadBucket` в `/health/ready` — единственная точка верификации что bucket существует и доступен.

### Runbook location and format
- **D-04:** Файл: `docs/runbooks/bucket-provisioning.md`. Новая папка `docs/runbooks/` создаётся под operational runbooks. Отделена от architecture docs (`docs/ARCHITECTURE_PRESENTATION.md`, `docs/TARGET_ARCHITECTURE.md`) и workflows (`docs/workflows.md`).
- **D-05:** Язык: русский. Согласуется с `docs/workflows.md`, `docs/TARGET_ARCHITECTURE.md`, `PROJECT.md`. Технические термины, env var names, команды, file paths остаются на английском inline.
- **D-06:** Один файл с разделами на окружение. Каждый раздел **self-contained** — читается линейно без "смотри там". Допускается общий Troubleshooting в конце если общих pitfalls много, иначе нет.

### Environment coverage
- **D-07:** Runbook покрывает ровно **четыре** окружения, каждое отдельным разделом в этом порядке:
  1. **Local-native** — `pnpm start:native` — infra в Docker (`docker-compose.infra.yml` + `docker-compose.dev-ports.yml`), apps на хосте. MinIO WebUI доступен на `http://localhost:9001`.
  2. **Local-isolated** — `pnpm start:isolated` — всё в Docker (`infra/docker-compose.yml`). MinIO WebUI так же на `localhost:9001` (порт тот же). Apps внутри docker network, не exposed на host.
  3. **Dev Coolify/Garage** — `garage.dev.email-platform.pp.ua` WebUI. Сервисы развёрнуты в Coolify dev environment. Buckets создаются в Garage, env vars устанавливаются в Coolify dev environment UI.
  4. **Prod Coolify/Garage** — `garage.email-platform.pp.ua` WebUI. Coolify prod environment. Те же шаги что в dev, только на prod target.
- **D-08:** Каждый раздел включает (в этом порядке):
  1. **Prerequisites** — что должно быть уже настроено (Docker запущен / доступ к Coolify / login к Garage)
  2. **Step 1: Open UI** — URL, login credentials source
  3. **Step 2: Create buckets** — точные имена (`parser`, `reports`), перечень
  4. **Step 3: Configure access** (для Garage: key + key binding; для MinIO: no-op или note что root user имеет полный доступ)
  5. **Step 4: Set env vars** — перечень требуемых `STORAGE_*` переменных, где их установить
  6. **Step 5: Verify** — `curl` команда к `/health/ready`, ожидаемый ответ, как понять что всё работает

### Bucket names (locked from code constants)
- **D-09:** Bucket `parser` — источник истины: `apps/parser/src/parser.constants.ts:6` (`PARSER_STORAGE_BUCKET = 'parser'`). Runbook цитирует эту константу inline (с file:line ссылкой) как "откуда берётся имя".
- **D-10:** Bucket `reports` — источник истины: `packages/foundation/src/external/storage/reports/reports.constants.ts:4` (`REPORTS_BUCKET = 'reports'`).
- **D-11:** Runbook открывается заметкой: _"Если имена bucket'ов в коде изменятся — обнови этот runbook в том же PR"_. Это единственная ручная sync точка между кодом и runbook.

### Environment variables documentation
- **D-12:** Runbook перечисляет полный набор `STORAGE_*` переменных из `packages/config/src/schemas/storage.ts`:
  - `STORAGE_PROTOCOL` (`http` | `https`)
  - `STORAGE_ENDPOINT`
  - `STORAGE_PORT`
  - `STORAGE_ACCESS_KEY`
  - `STORAGE_SECRET_KEY`
  - `STORAGE_REGION`
- **D-13:** Для каждого окружения runbook даёт **конкретные expected значения** (где возможно) или **placeholders** (где это секрет):
  - Local: `STORAGE_ENDPOINT=localhost` (native) / `minio` (isolated), `STORAGE_ACCESS_KEY=minioadmin`, `STORAGE_SECRET_KEY=minioadmin`
  - Dev/Prod: `STORAGE_ENDPOINT=<garage-endpoint>`, `STORAGE_ACCESS_KEY=<garage-access-key-from-coolify-secrets>`, `STORAGE_SECRET_KEY=<garage-secret-key-from-coolify-secrets>`
- **D-14:** Runbook ссылается на `.env.example:37-42` как reference для local значений.

### Garage-specific content
- **D-15:** Garage разделы (dev, prod) **отдельно** документируют шаги создания:
  1. bucket (через WebUI "Create bucket" или CLI `garage bucket create <name>`)
  2. access key (через WebUI "Create access key" или CLI `garage key create <key-name>`)
  3. **key binding** — привязка key к bucket (WebUI permissions page на bucket'е / CLI `garage key allow --read --write <bucket> <key-name>`)
- **D-16:** Без key binding bucket бесполезен — runbook **подчёркивает** это красным flagged блоком ("⚠️ БЕЗ ЭТОГО ШАГА BUCKET НЕ БУДЕТ ДОСТУПЕН"). Это самая частая ошибка при ручном setup.
- **D-17:** Основной путь — Garage WebUI (уже развёрнут на обоих окружениях согласно memory `project_hosting_infra.md` и `project_garage_s3.md`). **Отдельная секция "CLI alternative"** внутри каждого Garage раздела даёт эквивалентные команды `garage bucket create` / `garage key create` / `garage key allow` для тех кто предпочитает SSH/CLI подход (требует SSH доступа в Garage container на Coolify host).
- **D-18:** Runbook **не использует** S3 API CLI (`aws s3api create-bucket`) для Garage. Причина зафиксирована в rationale: S3 API не создаёт key binding, bucket окажется сломанным.

### Verification section
- **D-19:** Каждый раздел runbook заканчивается explicit verification step:
  - **Local-native:** `curl -s http://localhost:3003/health/ready | jq` (парсер на хосте)
  - **Local-isolated:** `docker compose -f infra/docker-compose.yml exec parser wget -qO- http://localhost:3003/health/ready` (парсер в docker, HTTP port не exposed)
  - **Dev Coolify:** `curl -s https://api.dev.email-platform.pp.ua/health/ready` (через gateway) + note что gateway gRPC health НЕ показывает S3 статус (см. D-21), для реальной S3 проверки нужен SSH в parser container
  - **Prod Coolify:** идентично dev, с `api.email-platform.pp.ua`
- **D-20:** Для каждого verification шага runbook показывает ожидаемый ответ (OK case) и пример failed ответа (503 с `s3:parser` / `s3:reports` в DOWN) чтобы оператор мог визуально сравнить.
- **D-21:** Runbook честно документирует **известный gap** в health mechanism:
  - Gateway `/health/ready` опрашивает backend-сервисы через **gRPC Health protocol** (`apps/gateway/src/health/health.controller.ts`)
  - Parser/notifier gRPC health возвращает **статичный `SERVING`** (`packages/foundation/src/external/grpc/grpc-server.factory.ts:13-15`)
  - Это означает что через gateway S3 проблемы **не видны** — gateway покажет `status: ok` даже когда S3 лежит
  - Реальная S3 верификация возможна только через прямой HTTP endpoint на parser/notifier
  - Fix этого gap — **отдельный concern, НЕ в скоупе 22.2**. Runbook упоминает этот факт как known limitation с ссылкой на файлы.

### Secrets handling
- **D-22:** Реальные credentials **никогда** не попадают в runbook. Placeholders используются везде:
  - `<GARAGE_ACCESS_KEY>` — берётся из Coolify environment secrets (указать точный путь: Coolify → project → environment → secrets)
  - `<GARAGE_SECRET_KEY>` — аналогично
  - `<COOLIFY_GATEWAY_URL>` — `api.dev.email-platform.pp.ua` / `api.email-platform.pp.ua`
- **D-23:** Для local: `minioadmin` / `minioadmin` указываются явно — это **известные публичные defaults для local dev**, не секреты.
- **D-24:** Runbook включает note: _"Для получения prod Garage credentials используй [тут путь] — не запрашивай в чате, не коммить в репо"_.

### Rationale section (embedded in runbook)

- Читается: (1) новыми разработчиками при onboarding, (2) DevOps/ops при настройке новых окружений, (3) phase 22.3 researcher'ом/planner'ом перед smoke tests, (4) будущим maintainer'ом которому придёт идея автоматизации.
- Единственная "live" связь с кодом — цитирование bucket name констант и `STORAGE_*` env vars (если константы меняются — runbook нужно обновить в том же PR, см. D-11).

### Known Gap (документируется в runbook, не исправляется)
- Gateway gRPC health protocol возвращает статичный `SERVING` для parser/notifier/всех backends — не видит реальный S3 state.
- Runbook объясняет что через `localhost:4000/health/ready` (local) или `api.*.email-platform.pp.ua/health/ready` (coolify) S3 проблемы не видны — нужен прямой HTTP endpoint на parser/notifier через `docker compose exec` (isolated) или SSH (coolify).
- Fix этого gap — **separate concern, не в скоупе phase 22.2**.

<specifics>
## Specific Ideas

- User прямо и окончательно подтвердил **полный отказ от автоматизации** в любой форме, включая docker-compose init containers. Предпочтение: unified manual approach с документацией.
- **Garage key binding concern** был ключевым триггером scope change: даже если S3 API `CreateBucket` работает на Garage формально, он не создаёт key binding, и бакет оказывается недоступным для приложения. Это делает кодовый provisioning бесполезным на production где используется Garage.
- **Философская позиция:** _"если код работает только в одном окружении — это code smell"_. Лучше ноль кода во всех окружениях (unified), чем код который включается только локально.
- **Knowledge capture motivation:** знание о setup buckets (какие buckets, какие имена, какие keys, какие env vars, где в Coolify) сейчас живёт только в голове одного человека. Runbook = явная архивация этого knowledge для всех будущих потребителей (новые разработчики, DevOps, будущий maintainer).
- **`docs/workflows.md` как стилевой template** — тот же tone (русский, technical, direct), та же структура (разделы по тематике, step-by-step instructions).
- **Self-contained чтение:** каждый раздел runbook должен быть читаемым независимо. Кто-то читающий только "Prod Coolify" должен получить полную картину без необходимости читать "Local-native" сначала.
- **Rationale section важна:** это _архив решения_ для будущего. Без неё через 6 месяцев кто-то (может быть сам user) подумает "ой, мы же забыли автоматизировать provisioning — давайте добавим". С ней — полный контекст почему решили наоборот и при каких условиях решение можно пересмотреть.
- **Visual warning для Garage key binding:** это самый частый failure mode при ручном setup, runbook должен подчёркивать этот шаг (bold / warning block / красный flag).

</specifics>

## Deferred Ideas

- **Автосоздание bucket'ов через кодовый `BucketProvisioningService`** — обсуждалось детально, отклонено. Причины: несовместимость с Garage key bindings, minimal permissions на prod (403 вместо 404 при HeadBucket), нарушение 12-factor principles, "код только для одного окружения" smell.
- **docker-compose `minio-init` container** для автоматизации local setup — обсуждалось, отклонено. User предпочитает unified manual approach по всем 4 окружениям, не хочет гибрид (auto local + manual prod/dev).
- **pnpm helper script (`pnpm setup:buckets`)** для local — обсуждалось, отклонено по той же причине unified approach.
- **Garage S3 API compatibility research** (Context7 / WebSearch / пробная установка) — могло бы разблокировать автоматизацию на Garage, но user не хочет исследовать сейчас. Если поменяется — runbook остаётся actionable reference, автоматизация может быть добавлена отдельной будущей фазой (например 999.X в backlog).
- **Fix gateway gRPC health sync с backing services** — runbook упоминает gap (D-21), но fix — отдельный concern, отдельная будущая фаза. Он касается gRPC health implementation в `grpc-server.factory.ts`, не bucket provisioning.
- **Screenshots в runbook'е** — overkill для первой версии. Текст + команды должно быть достаточно. Если в процессе написания runbook окажется что какой-то UI шаг требует визуализации — можно добавить локально, но не по умолчанию.
- **Structured troubleshooting section** — добавить только если в процессе написания runbook обнаружатся несколько общих pitfalls. Если нет — runbook остаётся без отдельного troubleshooting блока.
- **Automated runbook tests** (CI job который следует runbook шагам как integration test) — меганадёжно, но overkill для этого масштаба. Nice-to-have для будущего если runbook станет критическим для deploy process.
