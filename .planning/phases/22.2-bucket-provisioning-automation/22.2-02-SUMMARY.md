## Accomplishments

- **Полный operational runbook** — 765 строк, 10 top-level секций, 24 step headers (6 × 4 окружения), все acceptance criteria из плана пройдены
- **Четыре окружения, каждое self-contained** — local-native (MinIO WebUI localhost:9001), local-isolated (`docker compose exec` для non-exposed HTTP), dev Coolify/Garage (garage.dev.email-platform.pp.ua), prod Coolify/Garage (garage.email-platform.pp.ua). Каждое со стандартными 6 шагами: Prerequisites → Open UI → Create buckets → Configure access → Set env vars → Verify
- **Garage key binding safety** — оба Garage раздела включают highlighted warning блоки ("БЕЗ ЭТОГО ШАГА BUCKET НЕ БУДЕТ ДОСТУПЕН" в dev, "БЕЗ KEY BINDING BUCKET БУДЕТ НЕДОСТУПЕН" в prod), предотвращающие самую частую ошибку ручного setup. CLI alternative sub-sections с полными `garage bucket create` / `garage key create` / `garage key allow` командами для SSH-based workflow
- **Single source of truth для bucket names** — runbook цитирует `PARSER_STORAGE_BUCKET = 'parser'` из `apps/parser/src/parser.constants.ts:6` и `REPORTS_BUCKET = 'reports'` из `packages/foundation/src/external/storage/reports/reports.constants.ts:4` inline с file:line ссылками. Header-note напоминает обновить runbook в том же PR если константы изменятся
- **Полный перечень STORAGE_* env vars** — все 6 переменных из `packages/config/src/schemas/storage.ts` с concrete значениями для local (minioadmin/minioadmin открыто указаны как публичные dev defaults) и placeholder'ами для prod/dev (`<GARAGE_ACCESS_KEY>`, `<garage-dev-endpoint>`), с ссылкой на Coolify secrets как источник
- **Verification mechanism задокументирован** — каждое окружение заканчивается explicit curl/wget командой против `/health/ready`, с JSON примерами OK (HTTP 200) и DOWN (HTTP 503) ответов, s3:parser / s3:reports health keys объяснены. Documented HeadBucketCommand reference из `packages/foundation/src/internal/storage/s3.health.ts`
- **Known gateway gRPC health gap** — runbook честно документирует что gateway `/health/ready` опрашивает backends через gRPC Health protocol который в `packages/foundation/src/external/grpc/grpc-server.factory.ts:13-15` статично возвращает SERVING, поэтому S3 проблемы через gateway не видны — для реальной верификации нужен direct HTTP endpoint parser/notifier. Помечено как "separate concern, не в scope 22.2"
- **Rationale section как архив решения** — объясняет все 5 причин отказа от автоматизации: 12-factor separation, Garage key bindings incompatibility, Minimal permissions on prod, safety против silent misconfig, Unified approach по окружениям. Включает "Когда это решение стоит пересмотреть" sub-section с конкретными критериями для будущего maintainer'а
- **Quick matrix appendix** — сравнительная таблица всех 4 окружений (backend, endpoint, WebUI, verify command) для быстрого reference
- **Zero code impact** — ни одного файла под `apps/`, `packages/`, `infra/` не изменено (проверено через `git status --porcelain apps/ packages/ infra/`). Единственные изменения: `docs/runbooks/bucket-provisioning.md` (создан) + `docs/runbooks/` директория (создана)
