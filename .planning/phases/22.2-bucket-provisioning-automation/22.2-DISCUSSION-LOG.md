## Initial analysis

**Starting position:** Phase 22.2 originally scoped in ROADMAP.md as:
> _"Every bucket declared by a per-bucket module is automatically checked and created on service boot, idempotently and identically across MinIO and Garage, with failures surfaced through health checks"_

Requirements SPRV-01..05 anchored a code-based provisioning approach:
- SPRV-01: per-bucket module declaration as single source of truth
- SPRV-02: idempotent (HeadBucket → CreateBucket only on missing)
- SPRV-03: same code path for MinIO and Garage
- SPRV-04: provisioning errors block readiness (DOWN with structured reason)
- SPRV-05: opt-in via required env flag (config value, not env identity)

**Codebase scout surfaced:**
- `BucketStorageModule.forBucket({...})` as natural declaration point (Phase 22)
- `S3CoreModule` providing `S3_CLIENT` lazily (no network calls at construction)
- `S3HealthIndicator` already doing HeadBucket in readiness
- `S3ShutdownService` as existing lifecycle example in storage layer
- Zero existing `OnModuleInit` / `OnApplicationBootstrap` hooks in foundation storage
- Foundation partitioned into `internal/` (primitives) vs `external/` (facades) from Phase 22.1
- Services (parser, notifier) import per-service composition StorageModule, not raw S3CoreModule

## Lifecycle & trigger discussion

**Initial options presented:**

| Option | Description | Selected |
|--------|-------------|----------|
| OnApplicationBootstrap (Recommended) | После init всех модулей, перед listen(). S3Client гарантированно готов. Throw = процесс не стартует. Natural hard-fail boot gate. | ✓ |
| OnModuleInit | Во время init модуля, раньше, но зависит от порядка инстансиации. | |

**User's question before answering:** _"Тогда у нас не будет тогда наш механизм бесполезен, приложение стартует, пробует ли оно подключится к бакетам или нет? Если оно это делает и падает с ошибкой то мы не дойдем до нашего механизма который должен был создать бакеты?"_

**Claude's clarification:** S3Client — ленивый объект, никаких сетевых вызовов при конструировании или в module initialization. Первый реальный HeadBucket происходит либо:
1. В нашем `BucketProvisioningService.onApplicationBootstrap()` (если добавим)
2. В первом readiness request после `app.listen()`

Наш hook отрабатывает **первым** в любом случае. Никаких "автоматических" подключений до наших hooks не происходит. OnApplicationBootstrap безопаснее OnModuleInit потому что гарантированно срабатывает после init всех модулей (не зависит от порядка) и до listen().

**User's choice:** `OnApplicationBootstrap`.

**Note:** Этот выбор стал неактуальным позже после scope change (путь C: docs-only).

## Opt-in flag shape discussion

**Options presented:**

| Option | Description | Recommended | Selected |
|--------|-------------|-------------|----------|
| Глобальный STORAGE_PROVISION_ENABLED | Один boolean env var в StorageSchema, вся opt-in логика через него | ✓ | N/A |
| Per-bucket autoProvision параметр | forBucket({autoProvision: true}) — решение в коде | | |
| Комбо: global + per-bucket override | Env master + code override | | |

**Rationale for recommendation:** The global env var approach аligned со всеми skills:
- `env-schema/SKILL.md`: required env var без default, explicit `z.string().transform(v => v === 'true')`
- `twelve-factor/SKILL.md`: config value not env identity
- `branching-patterns/SKILL.md`: один guard clause (early return), без if/else chains
- SPRV-05 требует env flag

Per-bucket вариант был отклонён как нарушающий SPRV-05 (решение зашито в код, deployment не может переопределить без rebuild). Combo-вариант отклонён как YAGNI (нет use case для различного поведения разных buckets в одном сервисе).

**User's challenge:** _"Так нам нужна идемпотентность, то есть если на проде бакеты есть их не нужно создавать, зачем тогда флаг вообще? Или так не получится?"_

**Claude's response:** Очень важный вопрос. Idempotency (SPRV-02) и flag (SPRV-05) решают **разные** проблемы:

| Concern | Idempotency | Flag |
|---------|-------------|------|
| Защита от | Дублей при повторных стартах | Ненужных попыток вообще |
| Используется в | Dev, local, CI | Prod, Coolify, IaC-managed |
| Fallback | Если bucket есть — skip | Если flag=false — skip even HeadBucket |

Причины по которым flag нужен несмотря на idempotency:

1. **Права доступа (principle of least authority):** prod credentials часто write-only на уровне объектов, без `s3:CreateBucket` и даже `s3:HeadBucket` на уровне сервиса. Idempotent код вызовет HeadBucket → AccessDenied (403), не NotFound (404). Код не может отличить "нет прав" от "нет bucket" однозначно. С `STORAGE_PROVISION_ENABLED=false` ни один из этих вызовов не происходит.

2. **Garage key bindings (вероятная несовместимость):** Garage разделяет bucket и access key как отдельные сущности. Claude не уверен на 100% в Garage behavior через S3 API, но предположил что `CreateBucket` через S3 API создаёт bucket без key binding — access key не получает автоматически прав на новый bucket. Если так — кодовый provisioning на Garage создаёт сломанные buckets. С flag=false problem не возникает.

3. **Safety против misconfig:** если `STORAGE_BUCKET` переменная опечаталась (например `parser-dev` вместо `parser-prod` в prod .env), idempotent провижининг **тихо создаёт** wrong bucket и сервис работает с неправильным хранилищем. С flag=false HeadBucket на readiness громко падает — misconfig обнаруживается немедленно через 503 readiness.

4. **Ops/dev разделение ответственности:** в IaC/Coolify world buckets создаются admin'ом или deployment pipeline'ом, приложение только использует готовые. Flag=false формализует этот контракт.

## Scope change reasoning (KEY TURNING POINT)

**User's insight:** _"С такими вводыми нам вообще возможно стоит отказаться от автосоздания, то есть вносить в код приложения механизм, который будет делать это только локально, это через чур при том что это не предсказуемо, локально будет, в дев и прод не будет."_

**Claude recognized this as fundamental scope re-evaluation.** Three paths were presented:

| Path | Description | Trade-offs |
|------|-------------|------------|
| **Path A: code + init container** | Reject app code provisioning. Use docker-compose `minio-init` container для local. Prod/dev manual through Garage CLI. | Scope change. Cleaner 12-factor. Init container handles local automation. |
| **Path B: keep code provisioning with flag** | Original scope. Flag=true only local. Code exists but dormant in prod/dev. | "Code for one environment" smell. |
| **Path C: full manual + docs runbook** | No automation anywhere. Runbook covering all 4 environments. Knowledge capture. | Scope change. Radical but cleanest. |

**Claude recommended Path A initially** (init container as middle ground), but before presenting formal options added honest disclaimer about Garage behaviour uncertainty.

**User's follow-up question:** _"То есть и сейчас мне не нужно создавать никакие бакеты? Я еще просто не создавал, они создадутся сами?"_

**Claude clarification:** Currently nothing creates them. Neither init container nor app code. If parser/notifier services run now, `/health/ready` returns 503 with `s3:parser` and `s3:reports` down. To test locally, user would need to create buckets manually via MinIO WebUI or `mc` CLI.

**User's command to test health locally** led to clarification that:
- `start:isolated` mode exposes only gateway on `localhost:4000`
- Parser HTTP endpoint (3003) is not exposed (only expose, not ports in compose)
- Gateway `/health/ready` uses gRPC Health protocol which currently returns **static `SERVING`** — does not reflect S3 state (known gap in `grpc-server.factory.ts:13-15`)
- To verify S3 status, user must `docker compose exec parser wget -qO- http://localhost:3003/health/ready`

**User's final decision:** _"Ладно, сейчас это неважно. Мы не будем делать авто создание бакетов. Я создам их вручную, тогда нужно или скипнуть эту фазу или переделать ее в инструкцию как создать все нужные бакеты через минио интерфейс. И на прод и дев в garage на coolify. Что скажешь?"_

**Claude's response:** Strongly recommended Path C (docs) over skip, because:
1. Phase 22.3 (smoke tests) dependency on 22.2 remains valid if 22.2 becomes a runbook
2. Knowledge capture more valuable than emptiness
3. Artifact survives even if never automated
4. Soft-fallback for future: runbook can become reference if automation revisited
