    # Runbook: Создание S3 Bucket'ов

    > **Operational runbook** для ручной подготовки S3 хранилища во всех окружениях email-платформы.
    > Покрывает создание bucket'ов, access keys, key bindings (Garage), env vars и проверку работы.

    **⚠ Синхронизация с кодом:** если имена bucket'ов в коде изменятся, обнови этот runbook в том же PR.

    - `packages/foundation/src/external/storage/reports/reports.constants.ts:4` → `REPORTS_BUCKET = 'reports'`

    ## Что покрывает этот runbook

    Четыре окружения развёртывания в порядке возрастания критичности:

    1. **Local-native** — `pnpm start:native`, инфра в Docker, сервисы на хосте
    2. **Local-isolated** — `pnpm start:isolated`, всё в Docker
    3. **Dev Coolify/Garage** — `api.dev.email-platform.pp.ua` + Garage на `garage.dev.email-platform.pp.ua`
    4. **Prod Coolify/Garage** — `api.email-platform.pp.ua` + Garage на `garage.email-platform.pp.ua`

    Каждый раздел **self-contained** — его можно читать независимо от остальных. Если тебе нужен только prod — читай только prod.

    Каждый раздел следует одной и той же 6-шаговой структуре:

    |-----|-----------|
    | 1 | Prerequisites |
    | 2 | Open UI |
    | 3 | Create buckets |
    | 4 | Configure access |
    | 5 | Set env vars |
    | 6 | Verify |

    ## Bucket'ы и env vars

    **Два bucket'а** используются всеми окружениями одинаково:

    | Bucket | Имя | Источник истины | Используется |
    |--------|-----|-----------------|--------------|
    | `parser` | `'parser'` | `apps/parser/src/parser.constants.ts:6` | parser service — промежуточные файлы парсинга |
    | `reports` | `'reports'` | `packages/foundation/src/external/storage/reports/reports.constants.ts:4` | parser (запись) + notifier (чтение) — shared отчёты для Telegram |

    **Env vars** (полный список из `packages/config/src/schemas/storage.ts`):

    | Переменная | Тип | Пример local | Пример prod |
    |------------|-----|--------------|-------------|
    | `STORAGE_PROTOCOL` | `http` \| `https` | `http` | `https` |
    | `STORAGE_ENDPOINT` | hostname | `localhost` (native) / `minio` (isolated) | `<garage-endpoint>` |
    | `STORAGE_PORT` | number | `9000` | `443` |
    | `STORAGE_ACCESS_KEY` | string | `minioadmin` | `<GARAGE_ACCESS_KEY>` |
    | `STORAGE_SECRET_KEY` | string | `minioadmin` | `<GARAGE_SECRET_KEY>` |
    | `STORAGE_REGION` | string | `us-east-1` | `us-east-1` |

    Reference для local значений: `.env.example:37-42`.

    ## Что именно проверяется в readiness

    Readiness endpoint (`/health/ready`) parser'а и notifier'а выполняет `HeadBucket` против каждого привязанного bucket'а:

    ```typescript
    // packages/foundation/src/internal/storage/s3.health.ts:18
    await this.client.send(new HeadBucketCommand({ Bucket: this.bucket }));
    ```

    - **Parser** проверяет `parser` + `reports` — health keys `s3:parser`, `s3:reports`
    - **Notifier** проверяет `reports` — health key `s3:reports`
    - Если bucket не существует или credentials невалидны → `HeadBucket` падает → readiness возвращает 503 DOWN со структурированной причиной в `error` поле

    Это единственный механизм верификации что setup выполнен правильно. Не полагайся на "сервис стартанул" — сервис стартует даже без bucket'ов, но readiness будет DOWN.

    ## Known gap: gateway gRPC health не видит S3 state

    **Важно понимать перед verification:**

    Gateway `/health/ready` опрашивает backend-сервисы через **gRPC Health protocol**. Parser и notifier регистрируют статичный `SERVING` response при старте gRPC сервера:

    ```typescript
    // packages/foundation/src/external/grpc/grpc-server.factory.ts:13-15
    const healthImpl = new HealthImplementation({
      [HEALTH.GRPC_SERVICE_OVERALL]: HEALTH.GRPC_STATUS_SERVING,
    });
    ```

    Это означает: даже если S3 недоступен, gRPC health вернёт `SERVING`, и gateway'овый `/health/ready` покажет `status: ok`. **Через gateway S3 проблемы не видны.**

    Для реальной S3 верификации нужен **прямой HTTP endpoint** parser'а/notifier'а:
    - Local-native: `curl http://localhost:3003/health/ready` (parser), `curl http://localhost:3005/health/ready` (notifier)
    - Local-isolated: `docker compose exec parser wget -qO- http://localhost:3003/health/ready`
    - Coolify (dev/prod): SSH в parser/notifier container (HTTP port не exposed наружу через Traefik)

    Fix этого gap — отдельная задача, не в скоупе этого runbook'а. Runbook документирует как обходить gap через прямой endpoint.

    ## Почему не автоматизируем (Rationale)

    Этот runbook — ручная процедура. Code-based auto-provisioning (`BucketProvisioningService` или аналог который создаёт bucket'ы на boot сервиса) был рассмотрен и **отклонён**. Этот раздел — архив решения для будущего maintainer'а.

    **Пять причин:**

    ### 1. 12-factor separation (приложение vs инфраструктура)

    Приложение не должно управлять infrastructure state. Создание bucket'ов — это infrastructure concern (уровень оператора/IaC), не application concern. Код приложения должен **потреблять** уже настроенные ресурсы через env vars, не **создавать** их.

    См. `.agents/skills/twelve-factor/SKILL.md` — Factor IV (backing services as attached resources).

    ### 2. Garage key bindings несовместимы с S3 API `CreateBucket`

    Garage разделяет bucket и access key как независимые сущности. Создание bucket через S3 API `CreateBucket` **не создаёт key binding** автоматически — bucket получается изолированным и недоступным для приложения.

    Даже если бы мы написали код провизии — на Garage он создал бы **broken bucket** без key binding, и сервис всё равно упал бы на первой операции с `AccessDenied`. Key binding создаётся только через Garage-native API (`garage key allow`), который **не часть S3 API**.

    Единственный путь обойти это — Garage admin API интеграция в код приложения, что нарушает причину (1) ещё сильнее.

    ### 3. Minimal permissions на prod

    Prod credentials сервиса должны быть **минимальными**: read/write на конкретные objects в конкретных bucket'ах. Выдача `s3:CreateBucket` prod сервису — избыточный privilege escalation который усложняет audit и увеличивает blast radius компрометации.

    Код провизии на prod либо требует admin credentials (плохо), либо падает на `AccessDenied` при попытке `HeadBucket`/`CreateBucket` с write-only credentials — т.е. сервис boot'ится в broken state.

    ### 4. Safety против silent misconfig

    При кодовом provisioning'е опечатка в `STORAGE_BUCKET` name (например `repots` вместо `reports`) **молча создаёт wrong bucket** в prod, и сервис счастливо работает с неправильным хранилищем. Обнаружение — спустя часы/дни через "куда делись мои отчёты?".

    При manual approach + `HeadBucket` в readiness — любая опечатка моментально ловится через readiness DOWN на старте сервиса. Ошибка не доходит до production traffic.

    ### 5. Unified approach во всех окружениях

    Код который работает только в одном окружении (`flag=true` local, `flag=false` dev/prod) — code smell. Каждая ветка `if (provisionEnabled)` — это два разных поведения приложения, каждое со своими багами, каждое требует своего тестирования.

    Лучше **ноль кода провизии во всех окружениях** чем код который живёт только для local dev convenience.

    ### Когда это решение стоит пересмотреть

    Критерии при которых runbook стоит переcмотреть в пользу автоматизации:

    1. **Garage S3 API compatibility подтверждена** — если в будущем Garage начнёт автоматически создавать key binding при `CreateBucket` от owner key (или появится альтернативный S3-compatible backend с таким поведением), причина (2) отпадает.
    2. **Deployment переезжает на IaC** — если мы переходим на Terraform / Pulumi / Crossplane, provisioning переносится в IaC layer (не в код приложения) и причины (1), (3), (5) не применимы — IaC это **правильный** слой для infrastructure state.
    3. **Количество bucket'ов перестаёт помещаться в runbook** — если по какой-то причине у нас появится 20+ bucket'ов которые часто добавляются/удаляются, ручной процесс становится bottleneck'ом и рассмотреть IaC module.

    До тех пор — runbook остаётся единственным источником истины для bucket setup.
