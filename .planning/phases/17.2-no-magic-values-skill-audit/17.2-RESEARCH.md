# Phase 17.2: No Magic Values Skill & Audit - Research

**Researched:** 2026-04-04
**Domain:** Code quality / naming conventions / constant extraction
**Confidence:** HIGH

## Summary

This phase creates a new skill (`.agents/skills/no-magic-values/SKILL.md`) that classifies magic value violations and prescribes data structures for each case, then audits the entire codebase and fixes all violations according to the skill's rules.

The codebase already follows good practices in many areas -- `HEALTH`, `HEADER`, `SERVER`, `CORS`, `CONTEXT_TYPE`, `ERROR_MESSAGE`, `LOG_FORMAT`, `LOG_LEVEL` constants exist and are well-used. However, the audit uncovered several categories of remaining magic values: pool configuration numbers, DB schema varchar lengths, fallback strings in error filters, DI token strings in service modules, and `process.env.NODE_ENV` reads with `'development'` fallback strings that also violate 12-Factor rules.

**Primary recommendation:** Create the skill first with a clear decision tree, then fix violations in two waves: (1) packages/foundation constants and (2) apps-level constants. The skill should follow the branching-patterns SKILL.md structure exactly -- frontmatter, rule statement, decision tree, patterns with code examples, anti-patterns table, and review checklist.

```

### Anti-Patterns to Avoid

- **Bare numbers in provider factories** -- `max: 10` should be a named constant
- **Fallback strings inline** -- `'no-correlation-id'`, `'Internal Server Error'`, `'development'`
- **DI tokens as plain strings** -- `'UserRepositoryPort'` should be Symbol or `as const` string
- **VARCHAR lengths as bare numbers** -- `{ length: 255 }` repeated across schemas
- **`process.env.NODE_ENV || 'development'`** -- violates 12-Factor AND is a magic string

### Category 2: NODE_ENV Reads with Fallback (also 12-Factor violation)

| File | Line | Magic Value | Fix |
|------|------|-------------|-----|
| `logging.module.ts` | 40 | `process.env.NODE_ENV \|\| 'development'` | Remove NODE_ENV from base object entirely, or pipe through config |
| `logging.module.ts` | 92 | `process.env.NODE_ENV \|\| 'development'` | Same |

**Note:** Lines 134 and 193 in `forHttpAsync`/`forGrpcAsync` use `configService.get<string>('NODE_ENV')` which is slightly better but still reads NODE_ENV. ### Category 4: Retry Config with Direct process.env Reads and Fallbacks

| File | Line | Magic Value | Fix |
|------|------|-------------|-----|
| `retry-connect.ts` | 26 | `process.env.RETRY_MAX_RETRIES ?? ''` | Violates env-schema skill: should go through config module or keep RETRY_DEFAULTS only |
| `retry-connect.ts` | 29 | `process.env.RETRY_BASE_DELAY_MS ?? ''` | Same |
| `retry-connect.ts` | 32 | `process.env.RETRY_MAX_DELAY_MS ?? ''` | Same |

**Note:** These also use `||` fallback which violates the env-schema skill's "no fallbacks in consumer code" rule. However, RETRY_* vars are NOT in the Zod schema, so this is a design decision: either add them to the schema (preferred) or keep the defaults-only approach and remove the process.env reads entirely. Given the project constraint "no defaults in env schemas" and the fact that these retry values are infrastructure-level defaults that rarely change per environment, the cleaner fix is to REMOVE the process.env reads and keep only the RETRY_DEFAULTS object, allowing override only through function parameters.

### Category 5: DI Token Strings in Service Modules

| File | Token | Fix |
|------|-------|-----|
| `auth.module.ts` | `'UserRepositoryPort'`, `'LoginPort'` | Use Symbol tokens instead of strings |
| `sender.module.ts` | `'CampaignRepositoryPort'`, `'CreateCampaignPort'` | Same |
| `parser.module.ts` | `'ParserTaskRepositoryPort'`, `'StartParsingPort'` | Same |
| `audience.module.ts` | `'RecipientRepositoryPort'`, `'ImportRecipientsPort'` | Same |
| `notifier.module.ts` | `'HandleEventPort'`, `'NotificationSenderPort'` | Same |

**Note:** These are already exported as named constants (`export const USER_REPOSITORY_PORT = 'UserRepositoryPort'`), which is partially correct -- the value is named. However, the value is still a magic string, and using `Symbol()` is the project's own established pattern (see `DRIZZLE`, `PG_POOL`, `DATABASE_HEALTH` in persistence.constants.ts). Converting to Symbol ensures collision-free DI tokens. Each service should have its own `di-tokens.ts` or the tokens should live next to the port interface definitions.

### Category 6: Database Schema VARCHAR Lengths

| File | Values | Fix |
|------|--------|-----|
| `users.schema.ts` | `255`, `50` | Extract to `COLUMN_LENGTH.DEFAULT` (255), `COLUMN_LENGTH.SHORT` (50) |

### Category 7: Health Check / Bootstrap Strings

| File | Magic Value | Assessment |
|------|-------------|------------|
| `postgres.health.ts:17` | `'SELECT 1'` | Standard health check query -- extract to `PG_HEALTH_QUERY` constant |
| `postgres.health.ts:20` | `'PostgreSQL connection failed'` | Extract to health-related constant |
| `main.ts` (all 6 services) | `'Bootstrap failed:'` | Repeated across all services, extract to shared constant |
| `main.ts` (all 6 services) | `process.exit(1)` | The `1` is standard Unix exit code -- NOT a magic value, leave as-is |

### Category 8: Log Message Strings

| File | Magic Value | Assessment |
|------|-------------|------------|
| `grpc-logging.interceptor.ts` | `'OK'`, `'ERROR'`, `'gRPC call completed'`, `'gRPC call failed'` | Low priority -- log messages are not behavioral branching. However, `'OK'` and `'ERROR'` as status values should be extracted since they appear in structured log fields |
| `http-timing.interceptor.ts` | `'HTTP request completed'`, `'HTTP request failed'` | Same assessment |

**Recommendation:** Log status values (`'OK'`, `'ERROR'`) should be extracted to a `LOG_STATUS` constant. Log messages themselves are borderline -- they are human-readable strings, not identifiers. The skill should clarify that log message strings are ALLOWED inline because they are documentation, not behavior.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| DI tokens | String literals | `Symbol('TokenName')` | Collision-free, project convention |
| Repeated varchar lengths | Bare numbers per schema | Shared `COLUMN_LENGTH` constant | DRY, single source of truth |
| Pool config values | Inline numbers | Named constant object | Self-documenting, easy to find |

## Skill Design Specification

The skill MUST follow the exact structure of `branching-patterns/SKILL.md`:

### Required Sections

1. **Frontmatter** -- name, description, trigger keywords
2. **Rule** -- single clear prohibition with allowed/prohibited examples
3. **Decision Tree** -- flowchart mapping violation types to specific patterns
4. **Pattern sections** -- one per solution type with code examples
5. **Anti-Patterns table** -- prohibited patterns mapped to replacements
6. **Code Review Checklist** -- table of checks for PR review

### Decision Tree Design

```
Found a literal value in code?
|
+-- Is it a type literal in TypeScript? (type X = 'a' | 'b')
|   +-- ALLOWED -- type system, not runtime value
|
+-- Is it an import path or module specifier?
|   +-- ALLOWED -- structural, not behavioral
|
+-- Is it 0, 1, or -1 in array/loop/comparison?
|   +-- ALLOWED -- universal idioms (arr.length === 0, index + 1)
|
+-- Is it a log message string? (human-readable, not used as key)
|   +-- ALLOWED -- documentation, not identifier
|
+-- Is it process.exit(0) or process.exit(1)?
|   +-- ALLOWED -- standard Unix exit codes
|
+-- Is it a numeric config value? (timeout, pool size, retry count)
|   +-- Extract to named constant in *-constants.ts
|   +-- Use UPPER_SNAKE_CASE object: `const PG_POOL = { MAX: 10 } as const`
|
+-- Is it a string used as identifier/key/route/header?
|   +-- Used in one file only --> local `as const` object
|   +-- Used across files in same package --> package *-constants.ts
|   +-- Used across packages --> packages/foundation/src/constants.ts
|
+-- Is it a DI injection token?
|   +-- Use Symbol('TokenName') -- project convention
|
+-- Is it a set of related string values? (status codes, tiers, types)
|   +-- `as const` object with UPPER_SNAKE_CASE keys
|
+-- Is it a repeated number in DB schemas? (varchar length, precision)
|   +-- Extract to shared constant: COLUMN_LENGTH.DEFAULT
|
+-- Is it a fallback/default string? ('unknown', 'N/A', 'no-id')
    +-- Extract to named constant near its usage context
```

### Trigger Keywords for Frontmatter

`magic number, magic string, hardcoded value, literal, constant extraction, named constant, bare number, string literal, DI token, injection token, varchar length, timeout, pool size, retry count, exit code, fallback value`

## Common Pitfalls

### Pitfall 1: Over-Extracting
**What goes wrong:** Every single string and number gets extracted, including type literals, import paths, and idiomatic values like `0` and `1`.
**Why it happens:** Applying the rule mechanically without judgment.
**How to avoid:** The skill must have a clear "ALLOWED" section listing exceptions.
**Warning signs:** Constants named `ZERO`, `ONE`, `EMPTY_STRING`.

### Pitfall 2: Wrong Scope for Constants
**What goes wrong:** Service-specific constants placed in foundation package, or foundation constants duplicated per service.
**Why it happens:** Not thinking about ownership boundaries.
**How to avoid:** Decision tree includes scope guidance: "Used across packages?" determines where the constant lives.
**Warning signs:** Foundation importing from apps, or identical constants in multiple apps.

### Pitfall 3: Breaking Existing Imports
**What goes wrong:** Moving/renaming constants breaks imports elsewhere.
**Why it happens:** Not checking all usages before refactoring.
**How to avoid:** Use IDE/grep to find all references before changing. Verify with `pnpm turbo run build`.
**Warning signs:** Build failures after constant extraction.

### Pitfall 4: DI Token Symbol Migration
**What goes wrong:** Changing string tokens to Symbols breaks `@Inject()` decorators if not updated everywhere.
**Why it happens:** String tokens are compared by value, Symbols by reference. Must use the exact same Symbol instance.
**How to avoid:** Export Symbol from a single file, import everywhere. Run full build after migration.
**Warning signs:** NestJS "Cannot resolve dependency" errors at runtime.

export const DRIZZLE = Symbol('DRIZZLE');

export const PG_POOL_DEFAULTS = {

import { PG_POOL_DEFAULTS } from './persistence.constants';

const pgPoolProvider: Provider = {

  inject: [ConfigService],
  useFactory: (config: ConfigService): Pool =>

      max: PG_POOL_DEFAULTS.MAX_CONNECTIONS,
      idleTimeoutMillis: PG_POOL_DEFAULTS.IDLE_TIMEOUT_MS,
      connectionTimeoutMillis: PG_POOL_DEFAULTS.CONNECTION_TIMEOUT_MS,
    }),
};
```
