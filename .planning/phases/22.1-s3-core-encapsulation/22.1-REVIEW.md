# Фаза 22.1: Отчёт код-ревью

**Дата ревью:** 2026-04-09
**Режим:** standard
**Файлов отревьюено:** 16
**Статус:** issues_found

## Сводка

Фаза 22.1 `s3-core-encapsulation` перераспределяет S3-примитивы в `packages/foundation/src/internal/`, переворачивает публичный barrel на `external/`, добавляет `exports`-поле в `package.json` foundation с подпутём `./internal`, переключает `tsconfig.base.json` на `moduleResolution: node16`, поднимает ESLint-правило two-override против утечки `foundation/internal` в корневые модули и чинит DI-композицию `BucketStorageModule` так, чтобы она была самодостаточной (без `@Global()`).

Архитектурные цели выполнены корректно: три независимых ворот (exports field + node16 + ESLint) запечатывают разделение public/internal с защитой в глубину. Самодостаточность `BucketStorageModule.forBucket()` плюс переэкспорт `[BucketStorageModule]` по классу корректно распространяет Symbol DI-токены через динамический скоуп Nest 11. ESLint two-override pattern с явным пере-объявлением cross-app-ban в infrastructure-override корректно митигирует R-22.1-03 (без `'no-restricted-imports': 'off'`).

Найдена **одна реальная проблема** — `BucketStorageOptions.healthKey` объявлено в публичном контракте интерфейса, передаётся всеми вызывающими, но молча игнорируется внутри `BucketStorageModule.forBucket()`. Это dead data: контракт обманывает разработчика и разводит два источника истины для health-ключа (поле модуля + константа, передаваемая напрямую в `isHealthy()` из контроллера).

Остальные замечания — info-уровня: минорные стилистические непоследовательности, проглатывание потенциального undefined в factory-провайдере `S3CoreModule`, и несколько TODO-маркеров, унаследованных из предыдущих фаз.

## Warnings

### WR-01: `BucketStorageOptions.healthKey` — dead field в публичном контракте

**File:** `packages/foundation/src/internal/storage/storage.interfaces.ts:16-21`
**Related:**
- `packages/foundation/src/internal/storage/bucket-storage.module.ts:12-32`
- `packages/foundation/src/external/storage/reports/reports-storage.module.ts:12-18`
- `apps/parser/src/infrastructure/storage/parser-storage.module.ts:12-18`

**Issue:** Поле `healthKey` объявлено как обязательное в интерфейсе `BucketStorageOptions`:

```ts
export interface BucketStorageOptions {
  readonly bucket: string;
  readonly token: symbol;
  readonly healthToken: symbol;
  readonly healthKey: string;  // <-- объявлено
}
```

Оба потребителя обязаны его передавать (`REPORTS_HEALTH_KEY` и `PARSER_STORAGE_HEALTH_KEY`), TypeScript это проверяет. Однако внутри `BucketStorageModule.forBucket(options)` значение `options.healthKey` **нигде не считывается** — ни в провайдерах, ни в factory-функциях, ни в exports. `grep 'options\.healthKey' packages/foundation/src/internal/storage/` возвращает ноль совпадений.

Рантайм-значение health-ключа приходит в `S3HealthIndicator.isHealthy(key)` из второго источника — напрямую из app-кода (`REPORTS_HEALTH_KEY` / `PARSER_STORAGE_HEALTH_KEY`), переданного в health-контроллере:

```ts
// apps/parser/src/health/health.controller.ts:30-34
() => this.parserStorage.isHealthy(PARSER_STORAGE_HEALTH_KEY),
() => this.reportsStorage.isHealthy(REPORTS_HEALTH_KEY),
```

Таким образом каждый health-ключ имеет **два источника правды** (один в опциях модуля, второй в app-константе), и синхронизируются они только по глазному осмотру. Если в будущем кто-то отрефакторит `REPORTS_HEALTH_KEY = 's3:reports-v2'`, обновит использование в health-контроллере, но забудет константу в `forBucket()`, — расхождение пройдёт через build и lint молча, потому что значение в опции всё равно не читается.

**Fix:** Либо удалить поле из интерфейса (минимальный hard-cut), либо фактически его использовать. Рекомендуемый вариант — второй: пусть модуль регистрирует `healthKey` как встроенный провайдер и/или использует его при построении health-индикатора, чтобы контроллер получал ключ через DI, а не хардкодил его отдельной константой.

Минимальное исправление (удалить поле):

```ts
// storage.interfaces.ts
export interface BucketStorageOptions {
  readonly bucket: string;
  readonly token: symbol;
  readonly healthToken: symbol;
  // healthKey удалено — значение берётся из app-констант, передаваемых
  // напрямую в isHealthy() в health-контроллерах
}
```

с обновлением обоих call-сайтов (`reports-storage.module.ts`, `parser-storage.module.ts`) — убрать строку `healthKey: ...` из объекта опций.

Более архитектурно-цельное исправление (реально использовать поле) — дополнительно зарегистрировать провайдер `{ provide: options.healthToken, useFactory: ... }`, который сам знает свой ключ, и экспонировать pre-bound функцию, например `isHealthy()` без аргументов, либо выставить константу как provider под отдельным Symbol в том же модуле. Это более объёмная переработка; для текущей фазы достаточно hard-cut с удалением поля и TODO-записью в ROADMAP.

### IN-01: Non-null assertions на `ConfigService.get<T>()` в `S3CoreModule`

**File:** `packages/foundation/src/internal/storage/s3-core.module.ts:14-19`

**Issue:** Factory-провайдер `S3_CLIENT` читает env через цепочку non-null assertions:

```ts
const protocol = config.get<string>('STORAGE_PROTOCOL')!;
const endpoint = config.get<string>('STORAGE_ENDPOINT')!;
const port = config.get<number>('STORAGE_PORT')!;
const region = config.get<string>('STORAGE_REGION')!;
const accessKeyId = config.get<string>('STORAGE_ACCESS_KEY')!;
const secretAccessKey = config.get<string>('STORAGE_SECRET_KEY')!;
```

Согласно скиллу `env-schema` в `CLAUDE.md`, Zod-схема гарантирует обязательность всех env-переменных на этапе bootstrap, поэтому в рантайме эти значения не могут быть `undefined` — `!` семантически корректен. Тем не менее, подход хрупкий: если кто-то добавит новую переменную в `S3CoreModule`, но забудет внести её в `ParserEnvSchema`/`NotifierEnvSchema`, ошибка обнаружится только при старте сервиса. Более надёжный паттерн — использовать типизированную обёртку над `ConfigService.get()`, которая бросает явную ошибку, если ключа нет (и исключает `| undefined` из возвращаемого типа на уровне типов).

**Fix:** Рассмотреть на уровне всего foundation вынос в helper типа `requireEnv<T>(config, key): T`, чтобы заменить `!` на явную проверку с типовым сужением. Это out-of-scope для 22.1, но стоит зафиксировать в backlog.

### IN-02: Избыточный импорт `TerminusModule` в `NotifierModule`

**File:** `apps/notifier/src/notifier.module.ts:2`, `apps/notifier/src/notifier.module.ts:14-19`

**Issue:** `NotifierModule` импортирует `TerminusModule` напрямую:

```ts
import { TerminusModule } from '@nestjs/terminus';
// ...
imports: [
  AppConfigModule.forRoot(NotifierEnvSchema),
  TerminusModule,
  StorageModule,
  // ...
],
```

Начиная с Plan 05 `BucketStorageModule.forBucket()` сам импортирует `TerminusModule` в своём динамическом скоупе и экспортирует его наружу (`bucket-storage.module.ts:30`). Через цепочку `StorageModule → ReportsStorageModule → BucketStorageModule` `TerminusModule` уже становится доступен в скоупе `NotifierModule` транзитивно. Явный импорт `TerminusModule` в корневом модуле теперь дублирует этот транзитивный экспорт.

**Fix:** Удалить `TerminusModule` из `imports: [...]` и строку `import { TerminusModule } from '@nestjs/terminus'`. Верификация — `pnpm --filter @email-platform/notifier build && pnpm --filter @email-platform/notifier dev` + HTTP-запрос к `/health/ready` должен по-прежнему возвращать 200. Аналогичная проверка нужна и для `ParserModule`, хотя там явного импорта `TerminusModule` нет, так что изменений не требуется.

Альтернатива — оставить как есть, рассматривая явный импорт как документированный контракт («этот модуль использует Terminus»). Оба варианта приемлемы, но дублирование технически не нужно.

### IN-03: Несогласованность стиля в Symbol descriptions

**File:** `apps/parser/src/parser.constants.ts:1-4`, `packages/foundation/src/external/storage/reports/reports.constants.ts:1-2`

**Issue:** Описания `Symbol()` перемешивают два стиля именования:

```ts
// parser.constants.ts
export const PARSER_TASK_REPOSITORY_PORT = Symbol('ParserTaskRepositoryPort');  // PascalCase
export const START_PARSING_PORT = Symbol('StartParsingPort');                    // PascalCase
export const PARSER_STORAGE = Symbol('ParserStorage');                           // PascalCase
export const PARSER_STORAGE_HEALTH = Symbol('ParserStorageHealth');              // PascalCase

// reports.constants.ts
export const REPORTS_STORAGE = Symbol('REPORTS_STORAGE');                        // SCREAMING_SNAKE
export const REPORTS_STORAGE_HEALTH = Symbol('REPORTS_STORAGE_HEALTH');          // SCREAMING_SNAKE
```

Описания символов — отладочные, но глобально несогласованное форматирование усложняет чтение логов ошибок DI, где обычно печатается описание символа. В идеале одно и то же соглашение должно действовать на всю foundation + все apps.

**Fix:** Выбрать один стиль (рекомендуется SCREAMING_SNAKE, совпадающее с именем экспортируемой константы — так `toString()` символа совпадает с именем TS-переменной и поиск по логам тривиальный) и привести все символы к единому виду. Это out-of-scope для 22.1, но стоит зафиксировать в skill `no-magic-values` или создать отдельный backlog-item.

### IN-04: TODO-маркеры в `onModuleDestroy()` parser и notifier

**File:** `apps/parser/src/parser.module.ts:30`, `apps/notifier/src/notifier.module.ts:33`

**Issue:** Оба корневых модуля содержат явные TODO в хуке destroy:

```ts
// parser.module.ts:28-31
async onModuleDestroy(): Promise<void> {
  this.logger.log('Shutting down parser service...');
  // TODO: drain gRPC server connections
}

// notifier.module.ts:31-34
async onModuleDestroy(): Promise<void> {
  this.logger.log('Shutting down notifier service...');
  // TODO: close RabbitMQ subscriber connection
}
```

Это TODO не относятся к фазе 22.1 и унаследованы из предыдущей работы, но по регламенту ревью такие маркеры фиксируются как info. В теоретическом сценарии graceful shutdown под нагрузкой: активные gRPC-вызовы к `ParserGrpcServer` не получат chance дренажа, а активные сообщения RabbitMQ в `NotifierModule` не будут корректно завершены. S3-клиент, напротив, теперь корректно закрывается через `S3ShutdownService.onApplicationShutdown()` (это и есть часть достижений 22.1).

**Fix:** Out-of-scope для 22.1. Создать отдельный backlog-item на graceful shutdown gRPC-сервера и RabbitMQ-subscriber, привязать к фазе 25 (RabbitMQ event) или 23 (gRPC client).
