**Part A — Add the exports field to packages/foundation/package.json:**

Edit `packages/foundation/package.json` to insert the `exports` field. Place it after `"types"` and before `"scripts"`. The exact addition (preserving all other fields unchanged):

Before:
```json
{
    "name": "@email-platform/foundation",
    "version": "0.0.1",
    "description": "Shared NestJS foundation tools (gRPC clients, configs, metrics)",
    "main": "dist/index.js",
    "types": "dist/index.d.ts",

```

After:
```json
{
    "name": "@email-platform/foundation",
    "version": "0.0.1",
    "description": "Shared NestJS foundation tools (gRPC clients, configs, metrics)",
    "main": "dist/index.js",
    "types": "dist/index.d.ts",
    "exports": {
        ".": {
            "types": "./dist/index.d.ts",
            "default": "./dist/index.js"
        },
        "./internal": {
            "types": "./dist/internal/index.d.ts",
            "default": "./dist/internal/index.js"
        }
    },
    "scripts": {
        ...
    },
    ...
}
```

Key points:
- Two subpaths only: `"."` (public) and `"./internal"` (primitives)
- Each entry uses `types` + `default` conditions so TypeScript `.d.ts` lookup works under `moduleResolution: node16` (per RESEARCH.md Pattern 3)
- No `import`/`require` conditions — foundation emits CJS, no ESM consumers (per RESEARCH.md Pattern 3 rejected alternatives)
- No `./internal/*` wildcard — only the single barrel path is exposed (per RESEARCH.md Pattern 3 — keep the surface tight; wildcard can be added later if needed)
- `main` and `types` fields are RETAINED for compatibility with tooling that does not honor `exports` (some bundlers, older Node), even though `exports` takes precedence when both are present
