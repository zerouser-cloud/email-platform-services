## Decisions Made

- **Preserve the type-only `StorageHealthIndicator` re-export from Plan 02 (Rule 3 carry-forward):** Plan 04's literal file spec for `external/storage/index.ts` shows only `export * from './reports';`. The plan's `<interfaces>` snippet (lines 116-126) describes the Plan 02 "current" state as containing only `./reports` + the compat shim — it does NOT show the `export type { StorageHealthIndicator }` line that Plan 02 actually added as a Rule 3 auto-fix. Removing the type re-export would break the workspace build because `apps/parser/src/health/health.controller.ts` and `apps/notifier/src/health/health.controller.ts` both import `StorageHealthIndicator` as a type annotation for their injected `*_STORAGE_HEALTH` providers. Type-only exports compile away to nothing in CJS output (`tsc --emitDeclarationOnly` keeps them in .d.ts; runtime .js emits zero bytes for them), so they cannot leak any runtime primitive values — the SENC-02 invariant (S3CoreModule unreachable from `@email-platform/foundation` at runtime) is fully intact. The consequence is that the literal acceptance criterion `grep -c '^export '` = 1 is not satisfied (actual count: 2). All other acceptance criteria pass. The alternative — relocating the type to `external/storage/storage.interfaces.ts` or similar — is an architectural refactor that is out of scope for Plan 04 and could instead be scheduled for a future cleanup plan.
- **Atomic single commit (Part A + Part B bundled):** Per plan mandate and R-22.1-01 mitigation, splitting the subpath import switch from the compat shim removal would leave HEAD in an intermediate state where either (a) the shim is removed but parser-storage.module.ts still imports from the top-level barrel (build break), or (b) the subpath import is added but the shim still exposes BucketStorageModule publicly (defense-in-depth hole). Both changes land in `ae57788`.
- **Accept Turbo cache hits for this plan's build:** Plan 03 force-rebuilt the entire workspace with `cache bypass` so every task's hash was recomputed against the new tsconfig. Plan 04's edits trigger re-execution only for the packages that transitively depend on the edited files (`@email-platform/foundation` + `@email-platform/parser`); all other tasks are legitimate cache hits. Full workspace build still reports `10 successful, 10 total`.
