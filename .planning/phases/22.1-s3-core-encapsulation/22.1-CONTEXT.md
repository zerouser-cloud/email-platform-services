# Phase 22.1: s3-core-encapsulation - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Per-service composition StorageModule fully owns the S3 client lifecycle. Root service modules (`parser.module.ts`, `notifier.module.ts`) import a single per-service composition `StorageModule` and have no knowledge of underlying S3 infrastructure. Foundation src/ tree reorganized into `external/` (public facades) and `internal/` (primitives) so the encapsulation is enforced both at TypeScript/package-resolution level and via a single static ESLint rule.

Scope is the structural refactor only — no behavior changes, no new features, no API surface changes for storage tokens.

</domain>

<decisions>
## Implementation Decisions

### S3CoreModule DI strategy
- **D-01:** Remove `@Global()` decorator from `S3CoreModule`. S3 client visibility becomes explicit through standard NestJS module imports.
- **D-02:** `S3_CLIENT` remains a singleton in the NestJS DI tree — provider deduplication handles this regardless of how many modules import `S3CoreModule`.

### S3CoreModule import location
- **D-03:** Each per-bucket module (`ParserStorageModule`, `ReportsStorageModule`) imports `S3CoreModule` explicitly in its own `imports: [...]` array. NestJS sibling visibility does not propagate `S3_CLIENT` through a parent composition module, so each per-bucket consumer must declare its own dependency.
- **D-04:** `ReportsStorageModule` (in foundation) becomes self-contained — imports `S3CoreModule` itself and is plug-and-play. No caller setup required.
- **D-05:** Per-service composition `StorageModule` (in `apps/{service}/src/infrastructure/storage/storage.module.ts`) does NOT directly import `S3CoreModule`. It only imports per-bucket modules, which carry `S3CoreModule` transitively.

### Foundation package reorganization
- **D-06:** Foundation `src/` is split into two top-level directories: `external/` (public facades) and `internal/` (primitives that must be wrapped before use).
- **D-07:** `S3CoreModule`, `BucketStorageModule`, `S3StorageService`, `S3HealthIndicator`, `S3ShutdownService`, `storage.constants.ts`, `storage.interfaces.ts` move into `packages/foundation/src/internal/storage/`.
- **D-08:** `ReportsStorageModule` is a public facade — moves into `packages/foundation/src/external/storage/reports/`. It internally imports `S3CoreModule` from `../../../internal/storage` (relative path inside the package, no subpath import needed).
- **D-09:** All other current foundation modules (`logging`, `persistence`, `cache`, `errors`, `health`, `clients/grpc`, etc.) move from their current scattered locations into `packages/foundation/src/external/{module}/`. Their public APIs do not change in this phase.
- **D-10:** Top-level `packages/foundation/src/index.ts` becomes a one-line re-export: `export * from './external';`. This preserves backward compatibility with every existing `import { X } from '@email-platform/foundation'` call site.

### Subpath exports (package boundary)
- **D-11:** `packages/foundation/package.json` declares an `exports` field with two subpaths:
  - `"."` → public facades (`./dist/index.js`)
  - `"./internal"` → primitives (`./dist/internal/index.js`)
- **D-12:** Application code that attempts `import { S3CoreModule } from '@email-platform/foundation'` fails at compile-time because `S3CoreModule` is no longer reachable from the public barrel.
- **D-13:** `packages/foundation/tsconfig.json` (and any consumer tsconfig if needed) must use a `moduleResolution` that honors the `exports` field (`Bundler`, `Node16`, or `NodeNext`) — verify and adjust if currently using legacy resolution.

### Lint enforcement (single static rule)
- **D-14:** ONE `no-restricted-imports` rule is added in `.eslintrc.js`. It blocks `@email-platform/foundation/internal` and `@email-platform/foundation/internal/*` from files matching `apps/*/src/*.module.ts` (root service modules).
- **D-15:** ESLint override allows `@email-platform/foundation/internal` imports in files matching `apps/*/src/infrastructure/**/*.ts` (per-service infrastructure wrappers).
- **D-16:** The rule is **static** — never edited per phase. Adding any new internal primitive in future phases (Phase 23 gRPC channel, Phase 25 RabbitMQ raw connection, future) only requires placing the file under `packages/foundation/src/internal/...`. Zero `.eslintrc.js` changes needed.

### Refactor scope (this phase only)
- **D-17:** `parser.module.ts` and `notifier.module.ts` lose their direct `S3CoreModule` import. They keep importing only their per-service composition `StorageModule` (already exists from Phase 22).
- **D-18:** Existing per-service composition `StorageModule` files in `apps/parser` and `apps/notifier` remain in place — no rename, no relocation. Only their import statements change as needed (most will not need to change at all).
- **D-19:** `parser-storage.module.ts` and `notifier`'s wrapping for the reports bucket each gain one explicit `S3CoreModule` import (per D-03/D-04 above).
- **D-20:** All existing storage DI tokens (`PARSER_STORAGE`, `REPORTS_STORAGE`, `*_STORAGE_HEALTH`) and health controller integrations (`apps/parser/src/health/health.controller.ts`, `apps/notifier/src/health/health.controller.ts`) continue to resolve correctly post-refactor.
- **D-21:** `pnpm build` and `pnpm lint` must pass green at the end of refactor (success criterion SENC-04).
- **D-22:** No env var changes, no docker-compose changes, no port changes — pure code restructure.

### Naming
- **D-23:** Foundation top-level directories: `external` / `internal` (symmetric English antonyms, decided over `public/private` and `api/internal` for paired readability).

### Backlog items addressed structurally (informational, NOT in scope)
- **D-24:** This phase establishes the structural pattern that will close Phase 999.3 (PG_POOL leak) when it's promoted later — the fix is then "move PG_POOL into `internal/persistence/`", no special handling needed.
- **D-25:** Same pattern will apply to Phase 25 (raw RabbitMQ amqp connection), Phase 23 (raw gRPC channel/credentials), and any other future infra primitive — they all go in `internal/` automatically blocked from root by the same single rule.

### Claude's Discretion
- Exact `no-restricted-imports` pattern syntax (group vs paths form)
- Whether to use `package.json` `exports` field with conditions (e.g., `import` vs `require`) or simple subpath
- Internal subdirectory layout (`internal/storage/` flat vs nested, where `s3.health.ts` lives etc.)
- Whether the `external/` subdirectories follow the same per-concern shape as today (one folder per facade) or get further consolidated
- Build pipeline updates (`turbo.json`, `tsconfig.json` paths) needed to honor new dist output paths
- Order of operations during refactor (e.g., move foundation files first, then update imports, then add lint rule, then update root modules) — choose whatever keeps `pnpm build` recoverable between steps

</decisions>

<specifics>
## Specific Ideas

- Subpath-exports approach inspired by mature monorepos (Material UI, Next.js, React) where the public/internal API split is enforced via `package.json` `exports` rather than convention.
- User's stated architectural intent (use this as the principle for all future v4.0 phases): "все клиенты берут общую базу, каркас, инфру из foundation и конструируют свои клиенты, только их используют модули приложений" — root service modules consume only constructed clients, never raw infra primitives.
- Pair naming `external` / `internal` chosen as symmetric English antonyms — equally readable in both directions, and `internal` is the only conventional half of the pair.
- Lint rule and folder boundary chosen specifically to be **static** — adding new infrastructure primitives in future phases (gRPC channel, RabbitMQ raw connection, etc.) requires zero `.eslintrc.js` edits, only correct file placement under `internal/`.
- Phase 22 already left the per-service composition module scaffolding in place — Phase 22.1 is mostly relocation + one decorator removal + a handful of import-line changes.

</specifics>
