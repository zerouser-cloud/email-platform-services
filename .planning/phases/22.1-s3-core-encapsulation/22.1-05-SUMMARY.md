### Discovery (blocking) ‚Äî inherited DI regression from phase 22-03

First parser smoke attempt: `pnpm build && node apps/parser/dist/main.js` crashed at `NestFactory.create()` with:

```
Nest can't resolve dependencies of the ParserStorageModule
(exported from ParserStorageModule) that is not a part of the currently processed
module (ParserStorageModule). Please verify whether the exported Symbol(ParserStorage)
is available in this particular context.

UnknownExportException: Nest cannot export a provider/module that is not a part of the
currently processed module (ParserStorageModule).
```

**Root cause (determined via git archaeology):**

1. Phase 22-03 commit `f5f76b0 feat(22-03): add S3CoreModule global singleton` introduced `ParserStorageModule` / `ReportsStorageModule` with the shape:

   ```ts
   @Module({
     imports: [BucketStorageModule.forBucket({ token: PARSER_STORAGE, ... })],
     exports: [PARSER_STORAGE, PARSER_STORAGE_HEALTH],  // <-- exporting Symbol tokens
   })
   ```

2. This shape relied on `@Global()` on `S3CoreModule` to make `S3_CLIENT` reachable from the `BucketStorageModule.forBucket()` dynamic inner scope.

3. Plan 22.1-01 removed `@Global()` from `S3CoreModule` per SENC-01 ‚Äî correctly, per the plan spec.

4. Plans 22.1-01 through 22.1-04 added `S3CoreModule` as a sibling import in `ReportsStorageModule` / `ParserStorageModule` `imports: []` array. This ONLY makes `S3_CLIENT` visible to providers directly declared in the wrapper module ‚Äî NOT to providers declared inside `BucketStorageModule.forBucket()`'s dynamic inner scope. Nest DI does not cross sibling boundaries without `@Global()`.

5. Additionally, Nest 11 dynamic-module export resolution has a quirk: when a wrapper module imports a dynamic module and exports individual Symbol tokens declared inside that dynamic module, the reflected token array cannot find the providers via the wrapper's `reflectExports` path. The symptom is exactly `UnknownExportException: Nest cannot export ... Symbol(...) is available in this particular context`.

6. Static verification (build, lint, type check) DID NOT catch this regression ‚Äî it only surfaces at `NestFactory.create()` time during actual Nest DI graph resolution. Boot smoke was the first code path to trigger it.

7. `ReportsStorageModule` had structurally identical symptoms and would have failed identically on the notifier boot smoke after parser was fixed.

### Deviation ‚Äî 3-file atomic DI composition fix (commit `c906b1c`)

User approved the fix approach in chat after full evidence presentation:
- Git archaeology confirming phase 22-03 `@Global()` masking
- Analysis of Nest DI sibling-scope rules
- Mental model alignment via the user's own example code
- Explicit preference for preserving the `BucketStorageModule.forBucket({...})` dynamic factory pattern rather than rewriting to a helper function

**Fix (minimal ‚Äî 6 insertions, 7 deletions across 3 files):**

**1. `packages/foundation/src/internal/storage/bucket-storage.module.ts`** ‚Äî added `S3CoreModule` to `forBucket()` imports:

```diff
-import { TerminusModule } from '@nestjs/terminus';
+import { TerminusModule } from '@nestjs/terminus';
+import { S3CoreModule } from './s3-core.module';

 static forBucket(config: BucketConfig): DynamicModule {
   return {
     module: BucketStorageModule,
-    imports: [TerminusModule],
+    imports: [TerminusModule, S3CoreModule],
     providers: createBucketStorageProviders(config),

```

Now `BucketStorageModule` is self-contained: `S3_CLIENT` is resolvable within the bucket's own dynamic import chain. Multiple `forBucket()` calls share one `S3CoreModule` singleton via Nest's class-identity deduplication ‚Üí single `S3Client` instance across the application (not one per bucket).

**2. `apps/parser/src/infrastructure/storage/parser-storage.module.ts`** ‚Äî dropped redundant `S3CoreModule` import and switched to class re-export:

```diff
-import { BucketStorageModule, S3CoreModule } from '@email-platform/foundation/internal';
+import { BucketStorageModule } from '@email-platform/foundation/internal';

 @Module({
   imports: [
-    S3CoreModule,
```

Exporting `[BucketStorageModule]` by class (not `[PARSER_STORAGE, PARSER_STORAGE_HEALTH]` by Symbol) resolves the Nest 11 dynamic-module token-reflect quirk. Nest follows the class identity through the re-export chain, so downstream consumers (`HealthController`) still see `PARSER_STORAGE_HEALTH` through the re-exported dynamic scope.

**3. `packages/foundation/src/external/storage/reports/reports-storage.module.ts`** ‚Äî mirror change for `ReportsStorageModule`:

## Scope of boot smoke coverage (honest)

`HealthController` is the only code path in the project that currently `@Inject()`s storage tokens ‚Äî it injects `PARSER_STORAGE_HEALTH` and `REPORTS_STORAGE_HEALTH` (the health indicators). The service tokens `PARSER_STORAGE` and `REPORTS_STORAGE` (the actual `S3StorageService` instances) are registered by `createBucketStorageProviders` factories but are NOT `@Inject()`ed anywhere yet ‚Äî there is no upload/download/delete/exists/getSignedUrl code path in the current codebase.

Full CRUD round-trip validation across the five `StoragePort` methods is intentionally deferred to **Phase 22.3** (`storage-smoke-test-endpoints`) per roadmap ‚Äî that phase will add dedicated HTTP debug endpoints exercising the full `StoragePort` surface for every bound bucket on both parser and notifier, gated by a required env flag.

**However**, both the health tokens and the service tokens are created by **symmetric `useFactory` providers inside the same `BucketStorageModule.forBucket()` dynamic scope with the same `inject: [S3_CLIENT]` pattern** ‚Äî they share the identical DI resolution path. The health indicator round-trip (real `HeadBucketCommand` via AWS SDK `S3Client.send()`) exercises the same DI + transport path the service token would use on `putObject` / `getObject`. This indirect evidence, combined with the Phase 22.3 smoke-endpoints plan scheduled next in the roadmap, closes SENC-04 runtime-boot validation for Plan 05's scope.
