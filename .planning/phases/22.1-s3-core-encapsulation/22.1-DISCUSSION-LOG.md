## Area 1: S3CoreModule @Global() strategy

**Question:** Should `S3CoreModule` keep its `@Global()` decorator or have it removed?

| Option | Description | Selected |
|--------|-------------|----------|
| A. Keep `@Global()` | One import in composition `StorageModule`, NestJS DI does the rest. Convention-based encapsulation, less boilerplate. | |
| B. Remove `@Global()` | Each per-bucket module imports `S3CoreModule` explicitly. NestJS deduplicates singleton providers. Honest dependency graph, follows phase goal "fully owns the S3 client lifecycle" more literally. | ✓ |

**User's choice:** B. Remove `@Global()`

**Notes:** S3 client remains a singleton in NestJS DI tree regardless — provider deduplication is automatic.

## Area 2: S3CoreModule import location (after removing @Global)

**Question:** Where should `S3CoreModule` be imported now that `@Global()` is removed?

NestJS sibling visibility constraint: providers do NOT propagate from a parent composition module to its children. If composition `StorageModule` imports `[S3CoreModule, ParserStorageModule, ReportsStorageModule]`, neither per-bucket module can see `S3_CLIENT` through their parent. Each consumer of `S3_CLIENT` must declare its own dependency on `S3CoreModule`.

| Option | Description | Selected |
|--------|-------------|----------|
| I. Factory-level embed | `BucketStorageModule.forBucket()` embeds `S3CoreModule` in its returned DynamicModule. App code never sees `S3CoreModule`. Cleanest from app's POV, but wiring lives in foundation. | |
| II. Per-bucket explicit | Each per-bucket module (`ParserStorageModule`, `ReportsStorageModule`) sets `imports: [S3CoreModule, BucketStorageModule.forBucket(...)]`. `ReportsStorageModule` becomes self-contained and plug-and-play. | ✓ |

**User's choice:** II. Per-bucket explicit

**Notes:** ReportsStorageModule (in foundation) becomes plug-and-play — imports `S3CoreModule` from sibling internal/ files. No caller setup required.

## Area 3: Lint enforcement of encapsulation

**Question:** How should the encapsulation be enforced at root-module level?

Initial three-option presentation (A. specific S3-only ESLint rule, B. convention-only, C. generic infra rule with growing list) was rejected by the user. The user proposed a different mechanism: declare the boundary by **file location in foundation** rather than by maintaining a list in the lint rule. After clarification, the chosen mechanism is **package.json subpath exports + one static ESLint rule**.

### Refined approach

| Mechanism | Description | Selected |
|-----------|-------------|----------|
| Subpath exports + static rule | Foundation `src/` is split into `external/` (public) and `internal/` (primitives). `package.json` `exports` field declares `.` and `./internal` subpaths. ONE `no-restricted-imports` rule blocks `@email-platform/foundation/internal*` from `apps/*/src/*.module.ts`. ONE override allows it in `apps/*/src/infrastructure/**/*.ts`. Adding new primitives in future phases never requires editing `.eslintrc.js` — only correct file placement. | ✓ |
| Specific ESLint list (option A from initial presentation) | Hard-coded list of forbidden symbols (`S3CoreModule`, `BucketStorageModule`). Must be extended manually for every future infra primitive. | |
| Generic ESLint whitelist (option C from initial presentation) | Long regex of allowed symbols. Maintained in `.eslintrc.js`, requires updates per phase. | |
| Convention-only (option B from initial presentation) | Documentation + code review only, no enforcement. | |

**User's choice:** Subpath exports + static rule

### Naming sub-question

After agreeing on the structural approach, the user asked for symmetric folder naming. The user explicitly proposed `internal` paired with another name, and asked me to choose what works.

| Option | Description | Selected |
|--------|-------------|----------|
| `external` / `internal` | Symmetric English antonyms, user's original suggestion | ✓ |
| `public` / `internal` | Conventional API-design pair, slightly asymmetric | |
| `api` / `internal` | Common in some monorepos, less symmetric | |
| `facade` / `internal` | Describes architectural role | |

**User's choice:** `external` / `internal`

**Notes:** Foundation top-level `index.ts` becomes `export * from './external';` (one line). All current foundation files relocate under either `external/{module}/` or `internal/{module}/`. Backward-compatible — every existing `import { X } from '@email-platform/foundation'` still resolves through `external/`.
