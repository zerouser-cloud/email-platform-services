# Phase 22.1: s3-core-encapsulation - Research

**Researched:** 2026-04-09
**Domain:** NestJS DI structure + TypeScript package boundary enforcement (structural refactor, no behavior change)
**Confidence:** HIGH (stack verified against repo files + official docs + live npm metadata)

## Summary

Phase 22.1 is a **structural refactor** that closes a DI leak introduced by Phase 22. Today, `apps/parser/src/parser.module.ts` and `apps/notifier/src/notifier.module.ts` each import `S3CoreModule` directly from `@email-platform/foundation`, even though the per-service composition `StorageModule` at `apps/{service}/src/infrastructure/storage/storage.module.ts` is supposed to own the S3 client lifecycle. The leak was required because `S3CoreModule` is decorated with `@Global()` and is imported from the **top-level foundation barrel** тАФ so the only way to make its providers instantiate was to import it somewhere in the module tree of the root module. Phase 22 chose to do it at the root; Phase 22.1 pushes it down one level into the per-bucket modules and removes `@Global()`.

The refactor has four moving parts that must land together for `pnpm build` and `pnpm lint` to stay green: (1) NestJS side тАФ remove `@Global()`, add explicit `S3CoreModule` imports to `ParserStorageModule` and `ReportsStorageModule`, drop it from the two root modules. (2) Filesystem side тАФ reorganise `packages/foundation/src/` into `external/` and `internal/`, move every existing top-level subdirectory into `external/`, move the storage primitives into `internal/storage/`, keep the top-level `src/index.ts` as a one-line re-export. (3) Package boundary side тАФ add `exports` field to `packages/foundation/package.json` with two subpaths (`.` and `./internal`), and **upgrade `tsconfig.base.json`** from the legacy `moduleResolution: "node"` to `node16` (or `nodenext` / `bundler`) so the `exports` field is actually honored. (4) Lint side тАФ add one `no-restricted-imports` rule + one override block to `.eslintrc.js` enforcing that `apps/*/src/*.module.ts` cannot reach `@email-platform/foundation/internal`.

The research validated every decision in CONTEXT.md against current code, NestJS semantics, and TypeScript's module resolution rules. One issue requires user attention before planning proceeds: **D-13's `moduleResolution` upgrade is not a verification step тАФ it is a mandatory edit.** The current setting (`moduleResolution: "node"`) silently ignores the `exports` field, which would make D-11 and D-12 no-ops. The safest upgrade is `module: "node16"` + `moduleResolution: "node16"` on `tsconfig.base.json`. Because the foundation and app packages have no `"type": "module"` in their `package.json`, TypeScript will continue to emit CommonJS and existing relative imports without explicit `.js` extensions will keep working (CJS `require()` resolution does not require extensions, unlike ESM `import`).

**Primary recommendation:** Follow the six-step order-of-operations in ┬з"Architecture Patterns тЖТ Order of Operations" below. Every step keeps `pnpm build` green, and the ESLint rule is added last so it never fires on intermediate commits.

### Pattern 1: NestJS Singleton via Shared Module Import

**What:** When the same module is imported by multiple parent modules, NestJS's DI container treats it as a singleton тАФ providers are instantiated once.

**When to use:** Always. NestJS modules are singletons by default.

**Official statement:**
> "In Nest, modules are singletons by default, and thus you can share the same instance of any provider between multiple modules easily." [CITED: https://docs.nestjs.com/modules]

> "Every module is automatically a shared module. Once created any module can be reused by any number of modules." [CITED: https://docs.nestjs.com/modules]

**Applied here:**
```typescript
// packages/foundation/src/internal/storage/s3-core.module.ts
// NOTE: @Global() REMOVED compared to current state
import { Module } from '@nestjs/common';
// ...
@Module({
  imports: [ConfigModule],
  providers: [
    { provide: S3_CLIENT, inject: [ConfigService], useFactory: /* ... */ },
    S3ShutdownService,
  ],
  exports: [S3_CLIENT],
})
export class S3CoreModule {}
```

```typescript
// apps/parser/src/infrastructure/storage/parser-storage.module.ts
import { Module } from '@nestjs/common';
import { BucketStorageModule, S3CoreModule } from '@email-platform/foundation/internal';
// NOTE: S3CoreModule import ADDED here
import { /* tokens */ } from '../../parser.constants';

@Module({
  imports: [
    S3CoreModule,                                   // <-- NEW explicit import
    BucketStorageModule.forBucket({ /* ... */ }),
  ],
  exports: [PARSER_STORAGE, PARSER_STORAGE_HEALTH],
})
export class ParserStorageModule {}
```

```typescript
// packages/foundation/src/external/storage/reports/reports-storage.module.ts
import { Module } from '@nestjs/common';
import { BucketStorageModule, S3CoreModule } from '../../../internal/storage';
// NOTE: S3CoreModule import ADDED here via RELATIVE path (same package, no subpath)
import { /* tokens */ } from './reports.constants';

@Module({
  imports: [
    S3CoreModule,                                   // <-- NEW explicit import
    BucketStorageModule.forBucket({ /* ... */ }),
  ],
  exports: [REPORTS_STORAGE, REPORTS_STORAGE_HEALTH],
})
export class ReportsStorageModule {}
```

**Why this is safe (singleton preservation):** Both `ParserStorageModule` and `ReportsStorageModule` import `S3CoreModule`. The composition `StorageModule` at `apps/parser/src/infrastructure/storage/storage.module.ts` imports BOTH of them. NestJS recognises the same `S3CoreModule` class token and instantiates its providers (`S3_CLIENT`, `S3ShutdownService`) exactly once. Verified by official NestJS docs [CITED above] and by the existing project pattern тАФ Phase 22 already has `BucketStorageModule.forBucket()` imported twice (once by parser-storage, once by reports-storage) through a composition module, and `S3_CLIENT` is instantiated once.

### Pattern 2: Foundation Relative Imports (inside the package)

**What:** Inside `packages/foundation/src/`, modules use relative paths when crossing `external/` тЖФ `internal/`. No subpath import (`@email-platform/foundation/internal`) is needed inside the package itself.

**Why:** Subpath imports via `package.json` `exports` are only evaluated when TypeScript/Node resolves `@email-platform/foundation/...` from **outside** the package. Inside the package, the filesystem is the authority.

**Example:**
```typescript
// packages/foundation/src/external/storage/reports/reports-storage.module.ts
import { S3CoreModule, BucketStorageModule } from '../../../internal/storage';
// тЬЕ Works: relative path, three levels up (reports -> storage -> external -> src), into internal/storage
```

**Anti-example (DO NOT):**
```typescript
// Inside foundation
import { S3CoreModule } from '@email-platform/foundation/internal'; // тЭМ
// This creates a self-loop through node_modules symlinks and may not resolve during build.
```

### Pattern 3: Package.json `exports` Field тАФ Simple Subpath

**Recommended form (no conditions):**
```json
{
  "name": "@email-platform/foundation",
  "main": "dist/index.js",
  "types": "dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "default": "./dist/index.js"
    },
    "./internal": {
      "types": "./dist/internal/index.d.ts",
      "default": "./dist/internal/index.js"
    }
  }
}
```

**Why this form:**
- `types` condition is required so TypeScript `.d.ts` lookup works under `moduleResolution: node16/nodenext/bundler`. [CITED: https://www.typescriptlang.org/docs/handbook/modules/reference.html]
- `default` condition matches both `import` and `require` тАФ the foundation emits CJS (no `"type": "module"`) and consumers emit CJS, so a single `default` suffices.
- No `./internal/*` wildcard тАФ CONTEXT.md only requires `./internal` as a single barrel export. If a consumer ever needs deeper paths, add a pattern entry `"./internal/*": "./dist/internal/*.js"` in a later phase.
- Once `exports` is present, all subpaths NOT listed become unresolvable. That is the point тАФ it hides `@email-platform/foundation/internal/storage/s3-core.module` deep paths automatically. [CITED: "The presence of 'exports' prevents any subpaths not explicitly listed or matched by a pattern in 'exports' from being resolved." тАФ TypeScript handbook]

**Alternatives rejected:**
- Dual `import`/`require` conditions: not needed, no ESM consumers.
- `./internal/*` wildcard pattern: wider surface than necessary for this phase.

### Pattern 4: ESLint `no-restricted-imports` тАФ Patterns Block + Override

**New block to add in `.eslintrc.js` (top-level `rules` section):**
```javascript
rules: {
  // ... existing rules ...
  'no-restricted-imports': ['error', {
    patterns: [{
      group: [
        '@email-platform/foundation/internal',
        '@email-platform/foundation/internal/*',
      ],
      message: 'Internal foundation primitives cannot be imported from root service modules. Wrap in apps/{service}/src/infrastructure/ first.',
    }],
  }],
},
```

**New override block (under `overrides`) to allow the import in infrastructure:**
```javascript
{
  files: ['apps/*/src/infrastructure/**/*.ts'],
  rules: {
    'no-restricted-imports': 'off',
  },
},
```

**Why this shape:**
- `group` patterns use gitignore-style glob [CITED: https://eslint.org/docs/latest/rules/no-restricted-imports]. The two entries cover both the barrel (`@email-platform/foundation/internal`) and any future sub-path additions (`@email-platform/foundation/internal/*`).
- The override `'no-restricted-imports': 'off'` is the simplest approach. The existing `.eslintrc.js` already uses this pattern for the layered-architecture rules тАФ each override block re-defines `no-restricted-imports` scoped to the file set. Turning it off entirely in `apps/*/src/infrastructure/**/*.ts` is safe because no other `no-restricted-imports` rule currently targets files in that path (verified by reading `.eslintrc.js`).

**Anti-pattern (DO NOT):**
Do not target `apps/*/src/*.module.ts` with the restrictive rule using the `files` key on the top-level rule тАФ the top-level `rules` block does not accept `files`. The correct ESLint v8 pattern is: put the restrictive rule in top-level `rules` (applies everywhere by default), then use `overrides` blocks to relax it where allowed. Because no file outside `apps/*/src/` can legitimately import `@email-platform/foundation/internal` (packages/foundation/src/ uses relative imports inside itself; packages/config/src/ already cannot import foundation), this global-then-override approach is the simplest and safest.

**Ordering with existing `.eslintrc.js` rules:** ESLint's `overrides` blocks are applied in order. The last matching block wins. Because the top-level `rules` contains the new `no-restricted-imports`, and the override for `apps/*/src/infrastructure/**/*.ts` sets it to `off`, files in `apps/*/src/infrastructure/` will be allowed to import `@email-platform/foundation/internal`, while every other file (including `apps/*/src/*.module.ts`) will fail the check. Verified consistent with existing `packages/{contracts,config,foundation}/src/**/*.ts` overrides.

### Pattern 5: TypeScript `tsconfig.base.json` Upgrade

**Current (verified from tsconfig.base.json lines 1-29):**
```json
{
  "compilerOptions": {
    "module": "commonjs",
    "moduleResolution": "node",
    ...
  }
}
```

**Target:**
```json
{
  "compilerOptions": {
    "module": "node16",
    "moduleResolution": "node16",
    ...
  }
}
```

**Why node16 and not bundler or nodenext:**
- `node16` ensures TypeScript honors `package.json` `exports` field [CITED: https://www.typescriptlang.org/tsconfig/moduleResolution.html]
- With no `"type": "module"` in `packages/foundation/package.json` or any `apps/*/package.json`, TypeScript emits **CommonJS** under `module: "node16"` for `.ts` files тАФ same output as before. [CITED: "both ESM and CommonJS emit are supported, based on the detected module format of each file" тАФ TypeScript handbook]
- In CJS-emitted files, relative imports do NOT need explicit `.js` extensions тАФ the `require()` resolution algorithm handles extensionless paths. [CITED: "CommonJS `require()` calls don't enforce explicit extensions the same way ESM imports do" тАФ TypeScript modules reference]
- `nodenext` is a moving target (follows latest Node), and `bundler` + `commonjs` is explicitly "rarely what people want" per TypeScript PR #62320. `node16` is the stable paired setting.

**Risk:** None verified for existing code. The upgrade preserves CJS emission and does not add the explicit-extension requirement. The one concrete behavioral change is that TypeScript will now honor `package.json` `exports` fields in consumed packages тАФ which is exactly what we want, but could theoretically break a dependency that has a broken `exports` field. None of the current runtime dependencies (`@aws-sdk/client-s3`, `ioredis`, `drizzle-orm`, `pg`, `@nestjs/*`) have known `exports` field issues at the versions pinned.

**Consumer scope:** `tsconfig.base.json` is extended by `packages/{contracts,config,foundation}/tsconfig.json` and all six `apps/*/tsconfig.json`. The upgrade cascades to every package and app. This is **required**: per TypeScript's resolution model, the **consumer's** `moduleResolution` setting determines whether `exports` is honored when importing a dependency [CITED: "The conditions used for resolving package.json exports are determined by the moduleResolution setting" тАФ TypeScript handbook]. If only foundation's tsconfig upgraded, apps would still see `@email-platform/foundation/internal` as resolvable through legacy node10-style subpath lookup and the boundary would not be enforced.

### Anti-Patterns to Avoid

- **Adding the ESLint rule mid-refactor.** The rule would fire during intermediate commits and make `pnpm lint` red. Add it LAST.
- **Upgrading `tsconfig.base.json` before `package.json` has `exports` field.** No harm, but it also does nothing useful until `exports` is present.
- **Leaving the old `src/storage/` directory in place as a no-op pass-through.** Delete it in the same commit that completes the reorganisation тАФ otherwise developers get confused about which location is canonical.
- **Importing `@email-platform/foundation/internal` from inside `packages/foundation/src/`.** Use relative paths only inside the package. The subpath only works for consumers.
- **Deferring the `@Global()` removal.** If `@Global()` stays, the explicit imports in `ParserStorageModule` / `ReportsStorageModule` are redundant-but-harmless тАФ but the whole point of the phase (structural enforcement) depends on the removal. Do it in Step 2.
- **Using `overrides` with a `files: ['apps/*/src/*.module.ts']` restrictive pattern.** ESLint's override mechanism layers configs per-file; use the top-level rule for restriction and the override to RELAX, not the other way around. The alternative (restrict via override) still works but is more fragile because of override-ordering semantics.

## Common Pitfalls

### Pitfall 1: `moduleResolution: "node"` silently ignoring `exports` field

**What goes wrong:** You add the `exports` field to `packages/foundation/package.json` with the expectation that `@email-platform/foundation/internal` becomes restricted. But the apps still compile with `import { S3CoreModule } from '@email-platform/foundation'` working AND also `import { S3CoreModule } from '@email-platform/foundation/internal/storage/s3-core.module'` working тАФ because `moduleResolution: "node"` (the current setting) uses Node.js legacy algorithm that predates the `exports` field.

**Why it happens:** [CITED: "Package.json exports are only supported in --moduleResolution node16 and --moduleResolution nodenext. The older `node` (node10) resolution mode does not support this feature." тАФ TypeScript handbook]

**How to avoid:** Upgrade `tsconfig.base.json` to `moduleResolution: "node16"` in the same step where `exports` is added.

**Warning signs:**
- `pnpm typecheck` passes even though you wrote `import { S3CoreModule } from '@email-platform/foundation/internal/storage/s3-core.module.js'` (extension-hacked path)
- `node -e "require('@email-platform/foundation/internal')"` works but you never actually added the `exports` field

### Pitfall 2: Relative-extension requirement under `module: "node16"`

**What goes wrong:** You upgrade to `module: "node16"` + `moduleResolution: "node16"` and suddenly every file that does `import { foo } from './bar'` (no `.js` suffix) fails to compile with error TS2834.

**Why it happens:** Error TS2834 fires when TypeScript detects the emitted output is ESM. This happens if any `package.json` in the project contains `"type": "module"`, or if a file has `.mts`/`.mjs` extension. [CITED: "Relative import paths need explicit file extensions in EcmaScript imports when '--moduleResolution' is 'node16' or 'nodenext'." тАФ Total TypeScript]

**How to avoid:** Verify NO `package.json` in the project has `"type": "module"` before the upgrade. Verified for this repo:
- `packages/foundation/package.json` тАФ no `"type"` field [VERIFIED]
- `apps/parser/package.json` тАФ no `"type"` field [VERIFIED]
- (Other app package.json files need the same check during planning)

With no `"type": "module"`, TypeScript emits CJS under `module: "node16"` and CJS `require()` resolution does not require explicit extensions.

**Warning signs:** First `tsc` run after the upgrade throws a flood of TS2834 errors. If this happens, revert the tsconfig change and grep the repo for `"type": "module"` тАФ something in the workspace opted in without you noticing.

### Pitfall 3: Forgetting that `ReportsStorageModule` lives in foundation (not in a per-service infrastructure dir)

**What goes wrong:** You think `ReportsStorageModule` is a per-service wrapper like `ParserStorageModule`, and you put it under `apps/parser/src/infrastructure/storage/` or duplicate it in both parser and notifier.

**Why it happens:** The name pattern "XxxStorageModule" suggests a per-service wrapper. But `ReportsStorageModule` is actually a **shared facade** that encapsulates the shared `reports` bucket. It lives in foundation so both parser and notifier can import the same module and share the same DI token (`REPORTS_STORAGE`).

**How to avoid:** `ReportsStorageModule` moves to `packages/foundation/src/external/storage/reports/` (NOT to any `apps/*/src/infrastructure/`). Per D-08. It must add an explicit `S3CoreModule` import after `@Global()` is removed, because it's now responsible for pulling the S3 client into its own module graph. Per D-04.

**Warning signs:** A plan that puts `ReportsStorageModule` under `apps/` or uses `BucketStorageModule.forBucket({ bucket: 'reports', ... })` directly inside a per-service infrastructure file instead of importing `ReportsStorageModule` from foundation.

### Pitfall 4: Double-importing `S3CoreModule` at different levels

**What goes wrong:** You add `S3CoreModule` to `ParserStorageModule`, `ReportsStorageModule`, the composition `StorageModule` in `apps/parser/src/infrastructure/storage/storage.module.ts`, AND the root `parser.module.ts`. The code compiles and runs, but the encapsulation is broken.

**Why it happens:** Defensive programming instinct тАФ "add it everywhere to be safe." But this re-introduces the exact leak D-01..D-05 are closing.

**How to avoid:** Follow the decision matrix strictly:
- `S3CoreModule` import: ONLY in `ParserStorageModule` and `ReportsStorageModule` (the per-bucket modules)
- Composition `StorageModule` (in apps/{service}/src/infrastructure/storage/storage.module.ts): imports per-bucket modules ONLY, does NOT import `S3CoreModule`
- Root `parser.module.ts` / `notifier.module.ts`: imports composition `StorageModule` ONLY, does NOT import `S3CoreModule`

**Verification:** After the refactor, `grep -rn "S3CoreModule" apps/` must return exactly 1 hit per per-bucket module (1 in `parser-storage.module.ts`, 0 in per-service composition modules because `ReportsStorageModule` lives in foundation not apps, 0 in root modules). Add this grep to the Nyquist validation.

### Pitfall 5: Circular import between `external/storage/reports/` and `internal/storage/`

**What goes wrong:** `ReportsStorageModule` (in external) imports from internal. If anything in internal imports from external, a cycle forms.

**Why it happens:** Developer adds a utility function to `external/storage/reports/utils.ts` and then later finds `internal/storage/s3.health.ts` wants the same utility, so they import it from external.

**How to avoid:** The dependency direction is strictly one-way: **external тЖТ internal**. Never the reverse. Verified for this phase: `internal/storage/` contains only S3 primitives that don't need anything from external, and `external/storage/reports/reports-storage.module.ts` is the only external file that imports from internal. There are no other cross-boundary imports. The ESLint rule does not enforce this direction (only the app тЖТ foundation boundary); a `no-restricted-imports` rule on `packages/foundation/src/internal/**/*.ts` blocking `@email-platform/foundation` (the package name itself) would enforce it, but that is deferred per D-14..D-16 ("ONE rule, static").

**Warning signs:** `pnpm build` hangs or emits "Circular dependency detected" warnings. Any attempt to put shared utilities under `external/` that are imported by `internal/` files.

### Pitfall 6: `S3ShutdownService` lifecycle no longer firing

**What goes wrong:** After removing `@Global()`, `S3ShutdownService` (which is a provider of `S3CoreModule`) is still instantiated (per D-02 singleton semantics), but you worry that `OnApplicationShutdown` hooks might not fire because the module is no longer global.

**How to avoid:** NestJS lifecycle hooks fire for ALL instantiated providers regardless of `@Global()`. `@Global()` only affects **visibility** of exports, not provider lifecycle. [CITED: NestJS modules documentation тАФ `@Global()` makes the module's exports visible everywhere without explicit import, but "it is advised to register globally-scoped modules only once, generally by the root or core module"]

**Verification during validation:** Bring up parser (or notifier), send SIGTERM, check logs for the S3 client shutdown message. The `S3ShutdownService` currently calls `this.client.destroy()` (verified from s3-shutdown.service.ts). Add a log line if needed to confirm firing.

### Pitfall 7: Moving a subsystem but forgetting to update `src/index.ts`

**What goes wrong:** Foundation's top-level `src/index.ts` has 20+ `export * from './path'` lines (verified: lines 1-23 of current index.ts). You move `cache/` to `external/cache/` and update `src/index.ts` to `export * from './external/cache'` but forget another line. `pnpm build` goes red at the next Turbo run.

**How to avoid:** The order of operations in ┬з"Pattern 6" is designed to make each move independently green. But the top-level index.ts has many lines, so use a grep-based sanity check after each subsystem move:
```bash
grep -c "export \* from " packages/foundation/src/index.ts
# Should decrease by the expected number of moved subsystems, OR
# the old long-form index.ts should be fully replaced with a single `export * from './external';` line in Step 4.
```

**Warning signs:** `pnpm build` fails in a DOWNSTREAM package (an app) with `Module '"@email-platform/foundation"' has no exported member 'SomeType'`.

Verified by re-reading the current .eslintrc.js: all four existing overrides define `no-restricted-imports` тАФ the new override for `apps/*/src/infrastructure/**/*.ts` must appear AFTER the existing `apps/*/src/**/*.ts` override (which already defines `no-restricted-imports` for cross-app bans), otherwise it will be shadowed.
