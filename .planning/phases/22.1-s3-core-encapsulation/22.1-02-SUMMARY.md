## BucketStorageModule compat shim rationale

`apps/parser/src/infrastructure/storage/parser-storage.module.ts` currently does `import { BucketStorageModule } from '@email-platform/foundation'`. That import must continue to build after the top-level barrel flip in this plan, **before** Plan 03's `package.json` `exports` field + `tsconfig.base.json` upgrade land. The clean long-term fix — Plan 04 Task 1 — rewrites the import to `@email-platform/foundation/internal`, but that subpath only resolves after Plan 03.

**Solution:** `external/storage/index.ts` carries a single, explicitly-documented `export { BucketStorageModule } from '../../internal/storage';` line between the clearly-marked `=== TEMPORARY Plan 04 compat shim ===` / `=== END Plan 04 compat shim ===` comment banners. Plan 04 Task 1 must delete this block in the same commit that rewires `parser-storage.module.ts` — the compat shim is scoped to exactly one plan boundary, not a long-lived workaround.

`S3CoreModule` is NOT re-exported from the compat shim — only `BucketStorageModule`. Verified with `grep "S3CoreModule" packages/foundation/src/external/storage/index.ts` → 0 hits. This preserves SENC-02 for the primitive S3 client while keeping the build green at the plan boundary.

## Decisions Made

- **Type-only StorageHealthIndicator re-export from external/storage/index.ts (Rule 3 auto-fix):** The barrel flip in Task 2 Part D broke the build because `apps/parser/src/health/health.controller.ts` and `apps/notifier/src/health/health.controller.ts` both import `StorageHealthIndicator` as a type from `@email-platform/foundation`. This type lives under `packages/foundation/src/internal/storage/storage.interfaces.ts`; before Plan 02, it reached the public barrel via the Plan 01 compat shim's `export * from '../internal/storage'`. After the flip, that path no longer existed. Rather than reverting the flip or editing app-side imports (which would touch files outside the plan's scope), the fix exposes the interface as type-only from `external/storage/index.ts` — the interface is consumer-facing (it describes the contract for the `*_STORAGE_HEALTH` DI tokens that Phase 22 established), directly equivalent to `DatabaseHealthIndicator` which is already exposed from `external/persistence`. The implementation class `S3HealthIndicator` stays internal. This preserves SENC-02's spirit (S3 primitives encapsulated) while keeping the build green.
- **Atomic Task 2 commit:** Parts A through G all landed in one commit (`cb35e3a`) per Rule R-22.1-02 / R-22.1-02b mitigation — splitting the file moves, barrel flip, and root module cleanup would have left intermediate red build states. Verified by `git show cb35e3a --stat` showing 9 files changed in a single unit.
- **git mv for all 42 relocations:** Preserves rename history so future developers can `git log --follow` the history of any moved file back through Phase 22, 21, and earlier.
- **Transient top-level src/index.ts form after Task 1:** The Task 1 intermediate form references `./external/*` paths for non-storage subsystems while `./storage` still points at the Plan 01 compat shim. This keeps Task 1 independently green and preserves the clean task boundary while deferring the full barrel flip (plus `src/storage/` deletion) to Task 2's atomic unit.
